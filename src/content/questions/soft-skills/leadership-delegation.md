## 参考答案

### ✅ 核心要点

1. **授权的本质**：不是「把活分出去」，而是「让对的人在对的时机做对的决策」
2. **信任但有验证**：给空间，但建立里程碑检查点——放权不等于放任
3. **梯次授权模型**：从「我来做你看着」→「我们一起做」→「你做我看着」→「你做我放心」
4. **容错机制**：允许犯错的预算（时间和技术空间），这是成长的代价也是信任的成本
5. **最难的一步**：克服「我自己做更快更好」的心态——这是技术转管理最大的坑

### 📖 深度展开

**授权成熟度模型（Hersey-Blanchard 情境领导的改编版）：**

```
员工能力 × 意愿 矩阵：

┌──────────────────────────────────────────────────┐
│                                                    │
│  高能力 + 高意愿 → 授权（Delegate）                 │
│  "你来做，我信任你，有需要找我"                      │
│  → 只看结果，不管过程                               │
│                                                    │
│  高能力 + 低意愿 → 参与（Participate）              │
│  "我们一起想想怎么推进，你有什么顾虑？"              │
│  → 解决动力问题，找回热情                           │
│                                                    │
│  低能力 + 高意愿 → 教练（Coach）                    │
│  "我先示范一遍，你来试，我旁边看着"                  │
│  → 手把手教，频繁反馈                               │
│                                                    │
│  低能力 + 低意愿 → 指导（Direct）                   │
│  "按这个步骤做，每天跟我同步进展"                    │
│  → 明确指令，紧密跟进                               │
│                                                    │
└──────────────────────────────────────────────────┘
```

**实战案例：授权战斗系统重构**

> 背景：项目需要将战斗系统从 MVC 重构为 ECS 架构。团队里有个工作 3 年的中级开发小张，能力不错但没独立负责过架构级重构。
>
> **错误做法（直接甩手）：**
> "小张，战斗系统重构交给你了，一个月后给我结果。"
> → 结果：方向跑偏，一个月后推翻重来，士气崩塌。
>
> **正确做法（梯次授权）：**
>
> **第 1 周 — 共同设计阶段（Coach）**
> - 我和小张一起做系统分析，我画架构图
> - 产出：重构方案文档，两人共同评审
> - 我的角色：引导者，提出关键问题（"ECS 里怎么处理战斗事件的有序消费？"）
>
> **第 2-3 周 — 试点实施阶段（Participate）**
> - 小张负责核心战斗 Entity 的拆分，我负责 System 层框架
> - 每天 15 分钟同步，每周做一次 Code Review
> - 产出：核心战斗在 ECS 下跑通，可以联调
>
> **第 4-6 周 — 全面推进阶段（Delegate）**
> - 小张主导剩余模块迁移，带 1-2 个初级开发一起做
> - 改为每周一次 1on1，只看里程碑达成和风险
> - 我的角色：资源协调 + 向上汇报
>
> **结果：** 小张完成了 80% 的重构工作，在过程中成长为可以独立负责模块的骨干。我在这个期间释放出的时间做了渲染管线的优化。

**授权的 4 个边界（什么不能授）：**

| 维度 | 可以授权 | 不能授权 |
|------|----------|----------|
| 技术决策 | 模块内实现方案 | 架构级方向变更 |
| 人事管理 | 代码 Review 反馈 | 绩效评定、晋升决定 |
| 对外沟通 | 组内技术分享 | 跨部门的承诺和排期 |
| 风险处置 | 常见 Bug 修复 | 上线阻塞性事故 |

**心理建设：从「自己做」到「让别人做」**

```
技术骨干的思维：
  "这个我来做最快，跟他们解释的时间我自己都写完了"

管理者的思维：
  "这次我教他做，花2倍时间。但下次他就能独立做，我彻底释放了"

关键认知转变：
  你的产出 = 你直接做的事 + 你通过别人做的事
  随着团队规模扩大，后者的权重必须越来越大

衡量指标：
  - 你休假一周，团队产出是否正常？
  - 你离开后，团队是否能自主运转？
  - 你培养的人是否能独当一面？
```

### ⚡ 实战经验

- **授权前先评估承接能力**：一个从未独立做过模块的人，不能直接给他一个核心系统。先给一个边界清晰的小模块练手
- **游戏行业特殊性**：美术、策划、程序的工作方式差异很大，授权方式也要不同。给美术授权是给风格参考和时间，给程序授权是给接口定义和里程碑
- **别做「假授权」**：嘴上说"交给你了"，实际上每天微操盯进度——这比不授权更伤士气。如果你忍不住想插手，说明你授权时机不对或人选不对
- **建立安全网**：授权的同时告诉对方"如果遇到困难，随时找我，这不是失败而是及时反馈"。降低求助的心理门槛
- **公开认可，私下纠偏**：做对了在全团队面前表扬，做错了在 1on1 里复盘。公开批评会让所有人不敢接活
