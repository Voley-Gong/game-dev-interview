## 参考答案

### ✅ 核心要点

1. **对事不对人**：复盘是找系统性问题，不是追究责任。一旦变成追责，后续复盘就没人说真话了
2. **结构化流程**：回顾目标 → 还原过程 → 分析根因 → 提炼行动项 → 跟踪落地
3. **数据驱动**：用数据还原事实，而非凭记忆和感觉争论
4. **行动项必须闭环**：每个行动项有 owner、deadline、验收标准，下次复盘先回顾上次行动项
5. **频率比深度重要**：小复盘高频 > 大复盘低频。每两周一次 Sprint Retro 远比季度复盘有效

### 📖 深度展开

**复盘的四个阶段详解：**

```
阶段一：回顾目标
  - 我们当初的计划是什么？
  - 预期结果是什么？（交付范围、时间、质量标准）
  - 关键假设是什么？

阶段二：还原过程
  - 实际发生了什么？（用数据说话：燃尽图、Bug 趋势、延期天数）
  - 关键时间节点发生了什么？
  - 哪些预期之外的事件影响了项目？

阶段三：分析根因
  - 哪些做得好？（要保持）
  - 哪些做得差？（要改进）
  - 为什么？（连问 5 个 Why，找到根因而非表象）

阶段四：行动闭环
  - 具体、可执行的行动项（不是"加强沟通"这种废话）
  - 每个 action item 指定 owner 和 deadline
  - 下次复盘第一件事：回顾上次 action items 完成情况
```

**游戏行业典型复盘场景与案例：**

**场景 1：版本延期**

> **表象**：原定 3 月底版本，推迟到 4 月中。
>
> **5 Why 分析：**
> - 为什么延期？→ 策划在中期改了核心玩法
> - 为什么中期改玩法？→ 竞品上线了类似功能，老板要求调整
> - 为什么竞品影响这么大？→ 项目早期没有明确的核心竞争力定义
> - 为什么没定义？→ 立项阶段策划和制作人对游戏体验没有共识
> - 为什么没共识？→ 立项流程缺少"体验原型验证"环节
>
> **行动项**：
> 1. 下个项目立项必须制作可玩原型（Owner: 制作人 / Deadline: 下个项目 Kick-off）
> 2. 建立竞品监控机制，月会同步（Owner: 主策划 / 每月）
> 3. 核心玩法变更需走变更评审（Owner: 主程 / 两周内出流程文档）

**场景 2：线上事故复盘**

```
线上 P0 事故：热更导致 iOS 包闪退

时间线还原（关键）：
  10:00  发出热更
  10:05  玩家群开始反馈闪退
  10:12  运营确认问题，通知技术
  10:15  技术定位到热更资源格式不兼容 iOS
  10:20  决定回滚热更
  10:30  回滚完成，闪退率恢复正常

根因分析：
  - 直接原因：热更包没在 iOS 真机测试
  - 流程原因：热更缺少平台覆盖测试（只有 Android 测试）
  - 系统原因：没有热更的自动化回归测试

行动项：
  1. 热更必须在 iOS + Android 双平台测试（Owner: QA / 即日起）
  2. 搭建热更自动化回归脚本（Owner: 自动化测试工程师 / 2 周内）
  3. 热更发布改为灰度策略：先 5% → 观察 30 分钟 → 全量（Owner: 主程 / 1 周内）
```

**有效复盘 vs 无效复盘：**

| 维度 | 无效复盘 | 有效复盘 |
|------|----------|----------|
| 氛围 | 追责、甩锅 | 对事不对人 |
| 分析深度 | "沟通不畅" | 连问 5 Why 找到根因 |
| 数据 | 凭记忆和感觉 | 燃尽图、Bug 趋势、监控数据 |
| 行动项 | "加强沟通" | "每日站会增加风险同步环节，Owner: 张三" |
| 跟踪 | 说完就忘 | 下次复盘先检查上次行动项 |
| 频率 | 项目结束才做 | 每两周 Sprint Retro |

**不同复盘类型的适用场景：**

```
Sprint Retro（每 2 周，1 小时）：
  - 范围：单个迭代
  - 参与者：开发团队
  - 产出：2-3 个流程改进项

里程碑复盘（每个里程碑后，2-3 小时）：
  - 范围：一个里程碑周期
  - 参与者：核心团队 + 制作人
  - 产出：流程优化 + 风险预案更新

事故复盘（事故后 48h 内，1 小时）：
  - 范围：单个事故
  - 参与者：相关人员 + 主程
  - 产出：时间线 + 根因 + 防护措施

项目总结复盘（项目上线后，半天）：
  - 范围：整个项目周期
  - 参与者：全员
  - 产出：经验沉淀文档 + 组织级流程建议
```

### ⚡ 实战经验

- **复盘的文化建设比流程更重要**：如果团队不敢说真话，再完美的流程也是空架子。主程要先自我批评，创造安全氛围
- **行动项不超过 5 个**：一次复盘列出 20 个 action item = 一个都不会执行。聚焦最高价值的 3-5 个
- **游戏行业特别注意**：策划改需求导致的延期，复盘时不要变成"策划 vs 程序"的对立。关注"需求变更流程"这个系统性问题
- **记录"做得好的"**：复盘不是批斗大会。好的经验不沉淀，下次还会踩同样的坑。把成功经验也变成 SOP
- **工具辅助**：用 Miro/Figma 做可视化复盘（时间线、鱼骨图），比纯口头讨论效果好 3 倍
