## 参考答案

### ✅ 核心要点

1. **版本控制先行**：Git 分支策略、代码仓库结构、`.gitignore` 配置——第一天就要搞定
2. **CI/CD 尽早搭建**：第一天能自动构建 + 打包，而不是等第一个 milestone 才想起来
3. **规范同步建立**：代码规范 + CR 流程 + 提交信息规范，趁团队小的时候低成本推行
4. **架构留扩展空间**：核心架构决策做好模块解耦，但不要过度设计——YAGNI 原则
5. **文档随代码走**：架构决策记录（ADR）、接口文档、环境配置——从第一天就是活文档

### 📖 深度展开

**新项目前两周工程化清单（按优先级排序）：**

```
Week 1：基础设施（必须完成）
─────────────────────────────────
Day 1-2  版本控制 + 仓库结构
         ├── Git 仓库初始化
         ├── 分支策略确定（Git Flow / Trunk-based）
         ├── 项目目录结构（按模块/功能分层）
         ├── .gitignore（引擎生成文件、IDE 配置、构建产物）
         └── README.md（项目简介 + 环境要求 + 启动方式）

Day 3    引擎/框架搭建
         ├── 项目模板初始化
         ├── 核心系统骨架（事件/资源/UI/网络 管理器）
         ├── 配置系统（策划可编辑的数据表结构）
         └── Log 系统（分级日志 + 远程上报）

Day 4    CI/CD 流水线
         ├── 自动构建（GitHub Actions / Jenkins）
         ├── 自动打包（每日 build + 触发式打包）
         ├── Lint + 格式检查
         └── 构建结果通知（飞书/Discord 机器人）

Day 5    代码规范 + CR 流程
         ├── 编码规范文档（命名/结构/注释）
         ├── ESLint/Prettier 或引擎对应工具配置
         ├── PR 模板（描述/影响/测试）
         └── CR 规则（至少 1 人 approve、主程保留否决权）

Week 2：开发流程（逐步完善）
─────────────────────────────────
Day 6-7  热更新方案确定
         ├── 代码热更方案（Lua/JS/自研）
         ├── 资源热更方案（版本号/差异包/CDN）
         └── 热更测试流程

Day 8    性能基准建立
         ├── 目标机型确定（中端机基线）
         ├── Profiler 基准场景（空场景/战斗/UI）
         ├── 性能红线（帧率/内存/包体）
         └── 自动化性能监控

Day 9    测试框架
         ├── 单元测试框架搭建
         ├── 核心逻辑的测试用例（战斗公式/数据校验）
         └── 测试覆盖率目标（核心模块 > 80%）

Day 10   文档体系
         ├── ADR（Architecture Decision Records）
         ├── API 文档自动生成
         ├── 环境配置文档（新人 1 天内能跑起来）
         └── Wiki/Confluence 空间搭建
```

**关键架构决策：模块解耦的"洋葱模型"：**

```
         ┌───────────────────────────┐
         │       Game Logic Layer     │  ← 业务逻辑（经常变化）
         ├───────────────────────────┤
         │     Framework Layer        │  ← 框架层（事件/UI/数据）
         ├───────────────────────────┤
         │    Engine Adapter Layer    │  ← 引擎适配层（隔离引擎API）
         ├───────────────────────────┤
         │    Core / Utilities        │  ← 基础工具（数学/容器/Log）
         └───────────────────────────┘

原则：
- 上层依赖下层，下层不知道上层存在
- 引擎 API 调用全部走 Adapter 层（换引擎时只改这一层）
- 业务逻辑层不直接调用引擎 API
- 模块间通信走事件系统，不直接引用
```

**ADR（架构决策记录）示例：**

```markdown
# ADR-001: 使用 Cocos Creator 3.8 作为客户端引擎

## 状态：Accepted（2026-06-15）

## 背景
新项目是一款 2D 卡牌 + 轻度 3D 战斗的游戏，目标平台为 iOS/Android，
团队有 4 人有 Cocos 经验，2 人有 Unity 经验。

## 决策
选择 Cocos Creator 3.8。

## 理由
1. 团队 2/3 成员有 Cocos 经验，上手成本最低
2. 2D 渲染性能优秀，包体小（目标 < 100MB）
3. 原生支持热更新，符合项目快速迭代需求
4. 免引擎授权费，降低项目成本

## 代价
- 3D 渲染能力弱于 Unity，轻度 3D 战斗需要优化
- 社区生态不如 Unity 丰富，部分第三方 SDK 需要自行接入

## 复审
3 个月后评估：3D 战斗性能是否达标？SDK 接入成本是否可控？
```

**常见"第一天就该做但没做"的坑：**

| 遗漏项 | 后果 | 修复成本 |
|--------|------|----------|
| 没有统一的 Log 系统 | 上线后排查问题全靠猜 | 高（需要改所有模块） |
| 没有配置 CSV/JSON 校验 | 策划填错数据直接崩服 | 中（加校验工具即可） |
| 没有确定目标机型 | 测试在中端机卡得不行 | 高（可能需要改渲染方案） |
| 没有分支策略 | 3 个月后 merge 冲突地狱 | 极高（几乎无法修复） |
| 资源命名无规范 | 美术资源互相覆盖，找文件靠猜 | 中（规范 + 批量重命名） |

### ⚡ 实战经验

- **不要过度工程化**：新项目最大的陷阱是"先搭一个完美的框架再开始写业务"。第一周的基础设施够用就行，迭代完善。我见过团队花了 2 个月搭框架，结果项目被砍了——框架一行业务代码都没跑过
- **游戏行业特殊考量——热更新必须 Day 1 就确定方案**：客户端发版周期长（App Store 审核 1-3 天），如果热更新方案到项目中后期才考虑，往往需要大面积重构代码结构。代码热更（Lua/JS）和资源热更的差异巨大，影响整个代码架构
- **CI/CD 是最好的"技术债还款工具"**：每天自动构建一个可玩版本，策划和制作人可以随时看到进展。这比任何周报都有效——也让技术团队更早收到反馈
- **从技术到 PM 的视角转变**：以前觉得工程化是"技术问题"，转型 PM 后发现工程化其实是"协作问题"——CI/CD 让策划能每天看到版本，规范让新人能快速上手，文档让 PM 能理解技术约束。工程化体系的 ROI 不在代码层面，而在团队效率层面
