## 参考答案

### ✅ 核心要点

1. **提前预警**：发现风险信号时立即上报，不等 deadline 才说
2. **根因分析**：区分「偶发性延迟」和「系统性延迟」
3. **给出选项**：永远带着方案汇报，不只带问题
4. **重新基线**：调整里程碑，制定可执行的追赶计划
5. **复盘防复发**：延期处理完后必须做根因复盘，建立预防机制

### 📖 深度展开

**延期的三种典型模式与应对：**

| 延期类型 | 典型症状 | 应对策略 |
|----------|----------|----------|
| 需求蔓延型 | 策划不断加功能、需求文档频繁变更 | 冻结范围、走变更流程、砍非核心功能 |
| 估算失误型 | 任务拆解太粗、乐观估时、忽略联调 | 重新估算、加 buffer、细化任务粒度 |
| 资源瓶颈型 | 关键人员被抽调、多人等待同一模块 | 调整并行度、交叉备份、借调资源 |

**危机汇报的「三选一」框架：**

当你发现项目大概率要延期时，给管理层三个选项而非一个问题：

```
选项 A：保质量，延期 2 周
  - 影响：上线推迟 2 周，市场窗口收窄
  - 需要：市场侧配合调整宣发节奏
  - 风险：竞品可能抢先

选项 B：保时间，砍功能
  - 影响：首版缺少 X、Y 两个系统，改为后续版本更新
  - 需要：策划确认核心玩法不依赖这两个系统
  - 风险：玩家首印象可能不够完整

选项 C：保范围+时间，加资源
  - 影响：紧急招聘/外包 2 人，成本增加约 ¥X
  - 需要：HR 配合 + 1 周培训周期
  - 风险：新人融入成本高，可能拖慢现有节奏
```

**关键原则：让管理层做选择题，不做填空题。**

**每日站会中的风险信号识别：**

```
🚩 红色信号（需立即升级）：
  - 连续 3 天同一个人报同一个 blocker
  - 核心模块的进度连续未达标
  - 测试反馈的 bug 数不降反升
  - 加班成为常态而非例外

🟡 黄色信号（需关注但不升级）：
  - 某个任务估时偏乐观（实际超出 20%）
  - 跨模块联调出现接口不匹配
  - 第三方 SDK 文档与实际不符
  - 新人上手速度低于预期

✅ 绿色信号（正常推进）：
  - 每日完成的 story point 稳定
  - bug 收敛曲线符合预期
  - 自动化测试通过率稳定
```

### ⚡ 实战经验

- **永远不要在 deadline 前一周才说延期**：这是最严重的管理失职。发现苗头不对，30% 概率会延期时就应该预警
- **「追加人力」往往是加速的减速**：参考 Brooks 法则——给延期的项目加人只会让它更延期。除非瓶颈在新内容量产而非架构
- **砍功能有技巧**：砍掉「锦上添花」的功能，保留「核心体验闭环」。用 MoSCoW 法则（Must / Should / Could / Won't）做优先级
- **延期后做「无指责复盘」**：目标是改进流程而非追责。问「流程哪里出了问题」，不问「谁的错」
- **玩家沟通**：如果是已公布日期的项目延期，提前 2 周发公告。诚实的「我们想做得更好」比突然跳票好一百倍
