## 参考答案

### ✅ 核心要点

1. **项目分级**：按重要紧急程度给项目分级（S/A/B/C），资源向高优倾斜
2. **梯队授权**：每个项目要有独立负责人，你不能同时是所有项目的瓶颈
3. **时间切块**：按项目优先级分配时间块，高优项目占大头
4. **统一节奏**：多个项目的里程碑对齐节奏，避免"全是deadline周"
5. **风险透明**：建立多项目看板，让上级和团队都看到全局状态

### 📖 深度展开

**游戏公司多项目场景：**

> 你是客户端技术负责人，同时负责：
> - **项目A**：主力游戏版本大更新（S级，3个月后上线）
> - **项目B**：海外版适配和本地化（A级，持续进行）
> - **项目C**：新技术预研 / 引擎升级（B级，Q4 前出结论）
> - **项目D**：线上版本的 hotfix 和运营支持（日常维护）

**项目优先级矩阵：**

```
         紧急
          ↑
   项目D  │  项目A
   (运营)  │  (主力版本)
  ────────┼──────────→ 重要
   项目C  │  项目B
   (预研)  │  (海外版)
          ↓
         不紧急

资源分配建议：
- 项目A：40% 精力 + 50% 人力
- 项目B：25% 精力 + 25% 人力
- 项目D：20% 精力 + 15% 人力（日常响应）
- 项目C：15% 精力 + 10% 人力（弹性）
```

**精力分配策略——时间切块法：**

| 时间块 | 分配 | 说明 |
|--------|------|------|
| 上午深度时间（2h） | 项目A | 不开会，专注核心模块设计和代码审查 |
| 下午前半（1.5h） | 项目B | 跟进海外版进度，解决技术阻塞 |
| 下午后半（1h） | 项目D | 处理线上问题、运营需求评审 |
| 周五下午（2h） | 项目C | 技术调研、POC 实验、写预研报告 |
| 每天 30min | 全局 | 多项目看板更新、风险扫描 |

**多项目看板模板：**

```
┌─────────┬──────────┬──────────┬──────────┬──────────┐
│  项目   │  本周目标 │   状态   │   风险   │  负责人   │
├─────────┼──────────┼──────────┼──────────┼──────────┤
│ 项目A   │ 战斗模块  │ 🟢 正常  │ 无       │ 张三      │
│ (版本)  │ 联调完成  │          │          │          │
├─────────┼──────────┼──────────┼──────────┼──────────┤
│ 项目B   │ 日韩本地  │ 🟡 延迟  │ 翻译资源 │ 李四      │
│ (海外)  │ 化提测    │          │ 不足     │          │
├─────────┼──────────┼──────────┼──────────┼──────────┤
│ 项目C   │ 引擎渲染  │ 🟢 正常  │ 无       │ 王五      │
│ (预研)  │ 方案定稿  │          │          │          │
├─────────┼──────────┼──────────┼──────────┼──────────┤
│ 项目D   │ 日常运维  │ 🟢 正常  │ 周末活动 │ 赵六      │
│ (运营)  │          │          │ 服务器   │          │
└─────────┴──────────┴──────────┴──────────┴──────────┘
```

**梯队授权——你不可能事事亲为：**

```
你的角色：技术总控 + 资源调度

项目A ────→ 技术骨干 张三（主程级别，能独立做架构决策）
项目B ────→ 高级开发 李四（有海外项目经验）
项目C ────→ 资深开发 王五（技术热情高，爱钻研）
项目D ────→ 中级开发 赵六（踏实可靠，执行力强）

你的工作：
- 每周与各负责人 1on1（各30min）
- 参与项目A的核心技术评审
- 项目B/C/D 只参与关键决策和风险审核
- 对外统一代表技术团队沟通
```

**冲突场景处理：**

> 项目A突然出现严重 bug 需要你投入，同时项目B 海外版上线在即。
>
> 处理方式：
> 1. 评估 bug 影响面——是否能授权张三带队解决？
> 2. 项目B 上线——李四能否独立完成，还是确实需要你？
> 3. 如果两边的确都需要你——向制作人说明冲突，申请资源支援
> 4. **不要两头跑然后两边都做不好**

### ⚡ 实战经验

- **游戏行业最大陷阱**：运营需求永远紧急，但战略上项目A才是核心。守住高优项目的时间不被侵占
- **授权不等于放权**：授权给别人做，但你要建立 checkpoint 和验收标准。不闻不问等于失职
- **学会说不**：当上级给你加第四、第五个项目时，拿出多项目看板展示当前负荷，用数据说"不"
- **跨项目知识共享**：定期组织各项目负责人碰头，把一个项目踩的坑分享给其他项目，避免重复填坑
- **弹性预留**：永远不要把精力排满 100%，至少留 20% 应对突发事件（线上事故、人员离职等）
