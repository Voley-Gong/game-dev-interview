## 参考答案

### ✅ 核心要点

1. **先降温，后处理**：冲突激烈时不要急于判断对错，先让双方冷静
2. **分别沟通，再面对面**：先 1on1 了解各自立场，再组织三方会谈
3. **聚焦行为，而非人格**：讨论"发生了什么"，而非"你是什么样的人"
4. **找到共同目标**：用项目目标统一分歧，让大家回到"我们在同一条船"
5. **明确底线和后果**：调解不是无限妥协，要有明确的行为准则和升级机制

### 📖 深度展开

**游戏行业典型场景：**

> 策划组和程序组在需求评审会上爆发冲突。策划觉得程序"总是说做不了"，程序觉得策划"需求写不清楚还天天改"。双方在群里互相截图嘲讽，气氛降到冰点。

**错误处理方式 vs 正确处理方式：**

| 错误做法 | 正确做法 |
|----------|----------|
| 在群里公开批评双方 | 私下分别沟通，保全面子 |
| 直接搬出职位压人 | 用共同目标引导共识 |
| 各打五十大板和稀泥 | 明确各自问题，给出改进要求 |
| 拖着不管等它自然平息 | 主动介入，设定解决时间线 |
| 只听一方就下结论 | 充分听取双方，交叉验证事实 |

**调解五步法（详细流程）：**

**第一步：观察评估（1-2天）**
- 不要在冲突爆发的第一时间冲上去
- 观察冲突的范围（两人 vs 两组）、烈度（口头 vs 已影响工作）、持续时间
- 判断：是否需要你介入？有些小摩擦让团队自行消化反而更好

**第二步：分别 1on1（各30-45分钟）**

```
沟通话术示例：

开场（降低防御）：
"我最近注意到你和 XX 在需求评审上有些摩擦，我不是来追究谁对谁错的，
想了解一下你的感受和想法。"

倾听（不要急于评判）：
- "你觉得最近最让你不舒服的是什么？"
- "你认为对方做错了什么？"
- "如果换做你，你希望对方怎么做？"

引导反思（温和转折）：
- "你觉得在这个过程中，有没有自己可以改进的地方？"
- "你觉得 XX 那么做，可能的原因是什么？"

约定（建立预期）：
"我会也找 XX 聊聊，然后我们一起开个短会。你希望我在会上怎么协调？"
```

**第三步：分析根因**

常见根因：
- **目标不一致**：策划要快速迭代，程序要稳定架构 → 考核指标冲突
- **信息不对称**：策划不知道技术成本，程序不理解设计意图
- **历史积怨**：之前某次合作不愉快，一直没化解
- **性格差异**：有人直接，有人委婉，沟通风格不匹配
- **资源竞争**：两个组争夺同一批开发资源

**第四步：三方会谈（30分钟以内）**

```
会议结构：
1. 开场定调（2分钟）
   "今天这个会的目的是把之前的问题翻篇，找到后续合作的方式。"

2. 各自表述（各5分钟）
   每人说出自己的困扰，用"我感到..."句式而非"你总是..."

3. 寻找共识（10分钟）
   - "我们都希望项目按时上线"
   - "我们都希望玩家体验好"
   - 在共同目标上达成一致

4. 约定改进（10分钟）
   - 策划：需求文档提前 2 天给程序评估
   - 程序：不能做时提供替代方案，而非只说"做不了"
   - 双方：每周固定一次需求对齐会

5. 总结跟进（3分钟）
   "两周后我们回顾一下，看合作是否顺畅了。"
```

**第五步：持续跟进**
- 两周后分别 1on1 确认改善情况
- 如果没有改善，考虑调整团队结构或绩效考核方式
- 如果矛盾已影响项目交付，必须向制作人/总监汇报，不能护短

### ⚡ 实战经验

- **游戏行业特殊性**：策划和程序的矛盾是行业级问题，不是你团队独有。建立"需求评审流程"比每次调解更有效
- **不要当裁判**：你是管理者不是法官，强行判定对错会同时失去双方的信任
- **关注绩效影响**：如果冲突已导致进度延误或代码质量下降，必须果断处理，不能为了"和谐"牺牲项目
- **预防优于治疗**：团建、跨组 Pair Work、定期 Swap 岗位体验（让程序体验策划工作）能大幅减少冲突
- **极端情况**：如果某人是反复制造冲突的"毒性员工"，经过 2-3 次调解无效后，应考虑调岗或劝退
