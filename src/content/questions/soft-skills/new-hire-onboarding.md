## 参考答案

### ✅ 核心要点

1. **入职前就开始**：新人报到之前，环境和任务已经准备就绪
2. **结构化路径**：30/60/90 天计划，从「能跑起来」到「能独立交付」再到「能影响他人」
3. **导师制（Buddy）**：指定一位资深成员做技术导师，降低新人的沟通成本
4. **渐进式任务**：从 Good First Issue 到独立模块，逐步建立信心和熟悉度
5. **文化与融入**：不只是技术上手，还要帮助新人理解团队文化和协作方式

### 📖 深度展开

**新人 Onboarding 时间轴设计：**

```
入职前（-7 天）
  ├── 准备开发环境清单和权限申请
  ├── 创建新人专属 Onboarding 文档（项目架构、代码规范、开发流程）
  ├── 指定技术导师（Buddy）并沟通职责
  ├── 准备第一个练手任务（Good First Issue）
  └── 通知团队，安排欢迎午餐
       ↓
第 1 周：环境与基础
  ├── Day 1：环境搭建 + 项目跑起来 + 团队介绍
  ├── Day 2-3：阅读核心文档 + 跑通本地开发流程
  ├── Day 4-5：完成第一个小任务（修复一个 Bug 或加一个小功能）
  └── 目标：能独立编译、运行项目，提交第一个 PR/MR
       ↓
第 2-4 周：熟悉模块
  ├── 分配一个中等复杂度的模块任务
  ├── 导师每日 15 分钟 check-in，解答问题
  ├── 安排与各模块负责人 1on1（了解上下游依赖）
  ├── 参与代码评审（先只读评审，学习规范和风格）
  └── 目标：理解所在模块的架构和数据流，独立完成一个迭代任务
       ↓
第 2-3 月：独立贡献
  ├── 承担完整的模块开发任务
  ├── 参与技术方案评审，开始发表意见
  ├── 开始参与 CR（Code Review），评审他人的代码
  ├── 在团队技术分享会上做一次分享（教是最好的学）
  └── 目标：成为模块的 Owner 之一，能独立解决该领域的问题
       ↓
第 3 月+：持续成长
  ├── 90 天回顾：做得好的、需要改进的、未来目标
  ├── 制定个性化的成长路径
  └── 逐步扩展技术广度和业务深度
```

**新人 Onboarding 文档模板：**

```markdown
# 欢迎加入 [项目名] 团队！

## 🚀 快速开始
1. 开发环境搭建指南（一步步的详细教程）
2. 代码仓库地址和权限申请
3. 开发工具推荐和配置（IDE 插件、调试工具）

## 📖 必读文档
- 项目架构概览（附架构图）
- 代码规范和分支策略
- 开发流程（需求 → 开发 → CR → 测试 → 发布）
- 常见问题 FAQ

## 🗺️ 项目地图
- 核心模块和负责人（遇到什么问题找谁）
- 关键流程的数据流图
- 重要的第三方 SDK 和服务

## 🎯 第一个任务
- Issue 链接：[#xxx]
- 预期产出：[具体描述]
- 预估时间：[1-2 天]
- 导师：[名字]
```

**导师（Buddy）职责清单：**

```
┌──────────────────────────────────────────────────────┐
│ 导师不是「师父」，而是「引导者」                       │
├──────────────────────────────────────────────────────┤
│                                                      │
│  ✅ 做：                                              │
│  • 前 2 周每天 15 分钟 check-in                       │
│  • 第 3-4 周每周 2 次 check-in                        │
│  • review 新人的前 5 个 PR，给详细注释               │
│  • 带新人参加各类会议，讲解背景和暗号                 │
│  • 帮新人建立跨团队人脉（介绍给相关合作方）           │
│                                                      │
│  ❌ 不做：                                            │
│  • 手把手教写代码（新人要独立思考）                   │
│  • 替新人解决问题（引导他自己找答案）                 │
│  • 评价新人的「愚蠢问题」（没有愚蠢的问题）           │
│  • 把自己的技术偏好强加给新人                         │
│                                                      │
└──────────────────────────────────────────────────────┘
```

**不同级别新人的差异化策略：**

| 级别 | 前 2 周重点 | 导师参与度 | 任务难度 |
|------|-------------|------------|----------|
| 初级（0-2 年） | 环境熟悉、编码规范、基础架构理解 | 高（每日 check-in） | Bug 修复、小功能 |
| 中级（3-5 年） | 模块架构、业务逻辑、协作流程 | 中（每周 2-3 次） | 独立模块开发 |
| 高级（5+ 年） | 全局架构、技术决策上下文、团队风格 | 低（按需沟通） | 架构优化、跨模块重构 |
| 管理/架构岗 | 团队动态、历史决策背景、政治地图 | 融入式（引导 + 观察） | 技术规划和评审 |

**新人 Onboarding 的常见陷阱：**

```
陷阱 1：「扔文档」式 Onboarding
  现象：给新人发一个 wiki 链接就不管了
  后果：新人看不懂上下文，看一天还是不会做
  解法：文档 + 导师讲解 + 动手实践，三者缺一不可

陷阱 2：「上手太快」的错觉
  现象：新人第 1 周就独立做需求，看起来很顺利
  后果：第 2 个月才发现代码不符合架构规范，技术债爆发
  解法：前 5 个 PR 必须严格 CR，确保理解规范后再放宽

陷阱 3：「没有人管」的真空期
  现象：导师忙于自己的需求，新人几天没人管
  后果：新人产生被忽视感，开始怀疑团队和自己的选择
  解法：导师的 Onboarding 职任纳入绩效考核，确保时间投入

陷阱 4：「只关注技术」的融入偏差
  现象：技术上手很快，但不了解团队协作方式和文化
  后果：技术没问题但协作处处碰壁，3 个月后离职
  解法：安排跨职能的 coffee chat，帮助建立人际网络
```

### ⚡ 实战经验

- **游戏行业的特殊性**：游戏项目代码量巨大（动辄百万行），新人很容易迷失。建议为新人准备一份「阅读路线图」——从入口函数开始，沿着主循环到核心系统，标注哪些代码是历史遗留（别学），哪些是参考典范（多学）
- **第一个任务的黄金法则**：选一个真实但低风险的 Bug，修复后能在 1-2 天内合入。新人获得「第一天就贡献了代码」的成就感，比任何欢迎仪式都有用
- **Onboarding 是团队投资**：前 2 周导师的投入大约占其 30% 的工作时间。这是一笔投资，回报是 3 个月后一个能独立战斗的队友，而非 6 个月后还在问基础问题的新手
- **知识沉淀的飞轮效应**：新人 Onboarding 过程中发现文档缺失或不清晰的地方，让新人自己补全。这样新人变成了下一个 Onboarding 质量的受益者和推动者，文档越用越好
- **定期收集反馈**：每个新人入职满 1 个月和 3 个月时，做一次匿名 Onboarding 体验调查（NPS 式打分 + 开放反馈）。这是你持续改进 Onboarding 流程的最好数据源。团队的 Onboarding 质量每提升一个档次，人才流失率就会显著下降
