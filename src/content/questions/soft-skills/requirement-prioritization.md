## 参考答案

### ✅ 核心要点

1. **承认不可能全做**：资源永远不够，优先级排序是技术管理者的核心能力
2. **多维度评估**：用框架替代直觉，避免「谁嗓门大谁优先」
3. **价值对齐**：优先级必须与项目目标和商业目标挂钩
4. **透明决策**：排期逻辑要可见、可质疑、可调整
5. **定期复审**：优先级不是一成不变的，需要随信息更新动态调整

### 📖 深度展开

**常用优先级框架对比：**

| 框架 | 核心逻辑 | 适用场景 | 局限性 |
|------|----------|----------|--------|
| **RICE** | Reach × Impact × Confidence / Effort | 产品功能排序 | 游戏行业 Reach 难以准确估算 |
| **MoSCoW** | Must / Should / Could / Won't | 版本规划、MVP 定义 | 颗粒度粗，同级别内缺乏区分 |
| **加权评分** | 多维度打分加权求和 | 技术需求、多利益方场景 | 权重设置主观，需要校准 |
| **Kano 模型** | 基本需求 / 期望需求 / 兴奋需求 | 玩家体验设计 | 偏定性，难以直接出排期 |
| **ICE** | Impact / Confidence / Ease | 快速粗筛 | 过于简化，不适合重大决策 |

**游戏行业实战推荐：加权评分矩阵**

```
评估维度（根据项目阶段调整权重）：
┌─────────────────────────────────────┬──────┐
│ 维度                                │ 权重 │
├─────────────────────────────────────┼──────┤
│ 商业价值（对收入/留存的影响）        │ 30%  │
│ 玩家体验（对核心玩法感受的影响）      │ 25%  │
│ 技术风险（不做的后果有多严重）        │ 20%  │
│ 实现成本（人天投入）                 │ 15%  │
│ 战略价值（技术积累/架构演进）         │ 10%  │
└─────────────────────────────────────┴──────┘

每个需求打 1-5 分，加权求和得出优先级分数
```

**实战案例：版本规划中的需求排序**

场景：MMO 项目即将进入春节大版本开发，策划提了 12 个功能需求，但只有 8 周开发时间，最多能做 6 个。

```
需求清单与评分：

┌────┬────────────────────┬──────┬──────┬──────┬──────┬──────┬────────┐
│ ID │ 需求               │ 商业 │ 体验 │ 风险 │ 成本 │ 战略 │ 总分   │
├────┼────────────────────┼──────┼──────┼──────┼──────┼──────┼────────┤
│ R1 │ 春节限时活动       │  5   │  4   │  5   │  3   │  2   │ 4.25   │
│ R2 │ 新职业上线         │  4   │  5   │  3   │  1   │  4   │ 3.65   │
│ R3 │ 社交系统重构       │  2   │  4   │  4   │  2   │  5   │ 3.30   │
│ R4 │ 充值系统优化       │  5   │  2   │  4   │  4   │  1   │ 3.55   │
│ R5 │ 战斗手感优化       │  2   │  5   │  3   │  3   │  3   │ 3.40   │
│ R6 │ 新手引导重做       │  3   │  4   │  4   │  3   │  3   │ 3.55   │
│ R7 │ 公会战玩法         │  4   │  4   │  3   │  2   │  3   │ 3.35   │
│ R8 │ 性能优化专项       │  2   │  3   │  5   │  3   │  4   │ 3.30   │
│ ...│                    │      │      │      │      │      │        │
└────┴────────────────────┴──────┴──────┴──────┴──────┴──────┴────────┘

→ 取 Top 6：R1 > R2 > R4/R6 > R5 > R7
→ 砍掉的需求不是「不做」，而是「下个版本再评估」
```

**处理「所有需求都是 P0」的困局：**

> 制作人：「这些需求春节前必须全部上线。」
>
> 技术负责人（你）：「我理解这些都很重要。但 8 周只能做 6 个。我按照商业价值和风险做了评分，春节活动和新职业是最高优先级。如果都要做，我们看几个方案：」
>
> ```
> 方案 A：砍 2 个低优需求 → 风险可控，质量有保障
> 方案 B：全部做但加人 → 需要借调 3 人，沟通成本翻倍
> 方案 C：全部做不加人 → 预计延期 2 周，加班严重，质量风险高
> 方案 D：分两批上线 → 第一批春节前上，第二批元宵节上
> ```
>
> 「我推荐方案 D，商业上两波热度反而比挤在一起效果更好。您觉得呢？」

**「说不」的技巧：**

```
❌ 错误说法：
「做不了」「没时间」「这个不合理」

✅ 正确说法：
「这个需求我评估了一下，需要 15 人天。如果优先做它，
 R1 春节活动就会延期上线，影响 DAU 目标。
 我建议放到下个迭代，或者我们砍掉 R5 来腾出带宽。」

核心原则：
1. 给出成本估算（多少天/多少人）
2. 说明机会成本（做了A就不能做B）
3. 提供替代方案（而不是单纯拒绝）
4. 把决策权交还给需求方（让PM/制作人做选择）
```

**需求优先级的动态调整：**

```
每两周一次「需求复审会」（30分钟）：
  ├── 上周新增需求加入待评估池
  ├── 已有优先级是否需要调整（基于上线数据、竞品动态）
  ├── 紧急插入需求的处理（走「紧急通道」但需制作人签字）
  └── 更新需求看板，同步所有相关方
```

### ⚡ 实战经验

- **技术需求也要排序**：不只是产品需求，技术债清理、架构优化、性能改善也需要用同样的框架排序。技术管理者要主动把技术需求纳入整体优先级讨论，而不是偷偷塞进去
- **游戏行业特殊考量**：版本节点（节假日、赛季、IP联动）是硬约束。优先级排序必须围绕这些节点倒推，灵活调整权重。春节期间商业价值的权重可以临时提到 40%
- **数据驱动优先级**：「我觉得玩家需要这个」不如「留存数据显示 Day 3 流失集中在 35 级，我们需要补充这个等级段的内容」。推动团队用数据说话，减少主观争论
- **记住 70% 法则**：如果你 70% 确信一个需求的优先级判断是对的，就先执行。等待 100% 确信的成本远大于偶尔排错的成本。关键是定期复审、快速纠偏
