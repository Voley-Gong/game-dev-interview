## 参考答案

### ✅ 核心要点

1. **分层估算法**：粗估（T恤尺码）→ 细估（任务拆解）→ 精估（工时），精度递增、成本递增
2. **WBS 拆解原则**：任务粒度控制在 0.5-2 天，超过 2 天的任务必须继续拆
3. **三方估算法（PERT）**：乐观值(O) + 最可能值(M) + 悲观值(P)，公式：(O + 4M + P) / 6
4. **历史校准**：记录预估 vs 实际工时，定期修正团队"估算系数"
5. **留缓冲不留水分**：区分"任务时间"和"日历时间"，缓冲显式标注而非塞进每个任务

### 📖 深度展开

**估算方法对比表：**

| 方法 | 适用场景 | 精度 | 成本 | 推荐度 |
|------|----------|------|------|--------|
| 专家直觉法 | 熟悉模块、紧急评估 | ±40% | 极低 | ⭐⭐ |
| 类比法（找历史相似项） | 有类似项目经验 | ±30% | 低 | ⭐⭐⭐ |
| T恤尺码（XS/S/M/L/XL） | 项目初期粗排 | 量级正确即可 | 极低 | ⭐⭐⭐ |
| Planning Poker | 敏捷迭代估算 | ±25% | 中 | ⭐⭐⭐⭐ |
| WBS 逐项拆解 | 里程碑详细排期 | ±15-20% | 高 | ⭐⭐⭐⭐⭐ |
| PERT 三点估算 | 高风险/高不确定性任务 | ±15% | 中高 | ⭐⭐⭐⭐ |

**实战排期流程（WBS 法）：**

```
步骤 1：需求拆解
  需求文档 → 功能点列表 → 技术方案 → 任务清单

步骤 2：任务拆解（WBS）
  每个任务标注：
  - 预估工时（ideal hours，不含会议/中断）
  - 依赖关系（前置任务ID）
  - 风险等级（🟢低/🟡中/🔴高）
  - 负责人（或能力等级要求）

步骤 3：转日历时间
  日历时间 = 预估工时 ÷ 效率系数(0.6-0.7)
  （效率系数 = 纯编码时间占比，其余是会议/沟通/CR/上下文切换）

步骤 4：加缓冲
  - 个人级：高风险任务加 30% 缓冲
  - 里程碑级：总工期 × 15% 作为项目缓冲池
  - 缓冲由 PM/主程统一管理，不分配到个人

步骤 5：关键路径分析
  找出依赖最长的链路 → 这是项目真正的时间下限
```

**Planning Poker 实操（敏捷团队推荐）：**

```
准备：一套斐波那契数列卡片（1, 2, 3, 5, 8, 13, 21）

流程：
1. 产品经理/策划讲解一个需求（User Story）
2. 团队提问澄清（不超过 3 分钟）
3. 每人独立选一张卡片，背面朝上
4. 同时翻牌
5. 最高分和最低分的人各解释理由（2分钟each）
6. 再投一轮，通常就会收敛
7. 若仍分歧大（>2个数量级），拆分任务

关键好处：
- 避免锚定效应（资深成员先说，其他人跟风）
- 暴露隐藏的风险和理解偏差
- 团队对工作量达成共识
```

**常见估算偏差来源及对策：**

| 偏差来源 | 典型表现 | 对策 |
|----------|----------|------|
| 乐观偏差 | "这个功能半天就搞定了" | 要求写实现步骤，拆到0.5天粒度 |
| 忽略非编码时间 | 没算联调、测试、文档、CR时间 | 用日历时间 = 工时 ÷ 0.65 |
| 忘了依赖等待 | "UI给我我就开始" → UI晚了3天 | 排期图标注依赖，提前催 |
| 遗漏边界工作 | 忘了配置、部署、数据迁移、兼容性 | 准备"非功能任务清单"模板逐项check |
| 未考虑新人 ramp-up | 新人按熟手估 | 新人前两周效率系数按 0.3-0.4 |

### ⚡ 实战经验

- **永远不要给"一个数字"**：老板问"这个功能多久能做完"，回答应该是"最快2周，最可能3周，最慢5周，取决于美术资源是否就绪"。给区间比给点值专业十倍
- **游戏行业特有坑**：性能优化和适配（Android碎片化）是最大的估算黑洞。凡是涉及性能优化的任务，一律按 PERT 三点估算，悲观值往大了写
- **定期校准**：每个迭代结束后对比"预估值"和"实际值"，算出个人/团队的校准系数。坚持3个月，你的团队估算精度会显著提升
- **排期不是承诺**：区分"估算"（estimate）和"承诺"（commitment）。估算是技术判断，承诺是商业决策。不要把估算直接当承诺报上去——主程的职责是在两者之间翻译
