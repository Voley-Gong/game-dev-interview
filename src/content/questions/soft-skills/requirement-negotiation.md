## 参考答案

### ✅ 核心要点

1. **先理解「为什么」**：不要上来就否定方案，先搞清楚需求背后的真实目标
2. **用数据/成本说话**：把「做不了」变成「做这个的代价是 X，你确定要吗？」
3. **给替代方案**：拒绝的同时提供一个可行性更高的方案
4. **坚持底线但有弹性**：核心架构/性能底线不能让，但实现方式可以谈
5. **记录决策**：关键需求变更要有书面记录，口头 agreement 不算数

### 📖 深度展开

**需求谈判的四步模型：**

```
Step 1: 倾听 → 理解真实意图
  「我理解你想实现的是 XX 效果，对吧？」
  「这个需求是为了解决玩家的 XX 痛点？」

Step 2: 分析 → 暴露隐藏成本
  「这个功能涉及到 N 个模块的改动」
  「按这个方案做，中端机的内存会超红线」
  「开发周期约 X 天，会挤压 Y 功能的时间」

Step 3: 替代 → 提供更好选项
  「我有一个方案，用 Z 方式实现，只需要一半的开发量」
  「如果目标是 XX，其实用现有的 A 系统改造就能达到 80% 效果」

Step 4: 共识 → 让对方参与决策
  「我们目前有两条路：方案 A 成本高但效果完美，方案 B 快但有 X 限制」
  「你觉得哪个更适合当前版本？」
```

**场景模拟：策划要做一个复杂的实时天气系统**

```
❌ 错误回应（直接拒绝）：
  「这个做不了，性能扛不住。」
  → 策划觉得你在推活，矛盾升级

✅ 正确回应（理解+分析+替代）：
  「天气系统确实能提升沉浸感（肯定需求价值）。
   我看了一下，完整的实时天气模拟需要粒子系统+
   光照系统+物理系统的联动，大概 3 周工作量，
   而且中端机帧率可能掉 15fps（暴露成本）。

   我有两个替代思路：
   A) 用预渲染的天空盒 + 简单粒子，3 天搞定，
      视觉效果能达到 70%，性能几乎无影响
   B) 做 3 套预设天气状态（晴/雨/雾），切换式
      而非实时演算，1.5 周，效果 85%

   你觉得哪个方案更合适？或者我们聊聊哪些
   天气效果对你的玩法设计最重要？」
```

**「Yes, and...」替代「No, but...」：**

| 低效表达 | 高效表达 |
|----------|----------|
| 不行，这个做不了 | 可以，如果调整一下方案的话 |
| 没时间，排不上 | 可以排，但目前优先级排在 X 后面，大概 Y 时候能做 |
| 这个性能不行 | 这个方案在中端机上会掉帧，我建议改成 Z 方案 |
| 你不懂技术 | 我给你解释一下为什么这样做有风险 |

### ⚡ 实战经验

- **区分「需求」和「方案」**：策划给的是「方案」，他们的「需求」往往可以被更简单的方案满足
- **用策划听得懂的语言**：不要说「DrawCall 超了」，要说「同屏多 200 个动画物体，手机会卡」
- **POC 验证比口头争论有效**：与其在会上争论能不能做，不如花半天做个原型 demo，用事实说话
- **建立需求变更流程**：不是所有需求都要拒绝，但所有新增需求都要走评估流程。口头答应 = 无限蔓延
- **维护信任账户**：平时多帮策划解决小问题，关键时刻你拒绝大需求时，他们更愿意信任你的判断
