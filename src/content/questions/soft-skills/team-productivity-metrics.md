## 参考答案

### ✅ 核心要点

1. **效能 ≠ 效率**：效能是「做正确的事」，效率是「正确地做事」。先保证方向对，再优化执行速度
2. **多维度度量**：任何单一指标都会被博弈（Goodhart法则），至少用 3-5 个互补指标
3. **趋势 > 绝对值**：关注变化趋势而非单点数据，周/月对比比季度平均更有指导意义
4. **度量用于改进而非考核**：一旦指标与绩效强绑定，团队就会优化指标本身而非真正目标
5. **建立改进闭环**：度量 → 分析 → 行动 → 复验（PDCA循环）

### 📖 深度展开

**游戏客户端团队效能指标体系：**

| 维度 | 指标 | 采集方式 | 健康范围 | 警戒信号 |
|------|------|----------|----------|----------|
| **交付速度** | Sprint 速率（故事点/人/周） | Jira 自动统计 | 稳定或缓慢上升 | 连续3个Sprint下降 |
| **质量** | 千行Bug率（Bug数/KLOC） | Bugzilla + cloc | < 5 | > 10 需要专项治理 |
| **质量** | 线上Crash率 | Bugly/Crashlytics | < 0.1% | > 0.5% 立即响应 |
| **效率** | PR 平均合并时长 | Git 统计 | < 24小时 | > 48小时说明CR瓶颈 |
| **效率** | 构建/部署频率 | CI 统计 | 每日至少1次 | 每周 < 3次说明流程有问题 |
| **健康度** | 加班时长占比 | 工时系统 | < 10% | > 20% 有燃尽风险 |
| **健康度** | 代码review覆盖率 | Git + 脚本 | > 90% | < 70% 说明CR流于形式 |
| **技术债** | TODO/FIXME 密度 | 代码扫描 | < 2/KLOC | 持续增长需专项处理 |

**PDCA改进闭环实操：**

```
Plan（计划）—— 周五复盘会
  发现问题：PR合并时长 > 48小时，阻塞开发
  根因分析：CR 堆积在周五，2名高级工程师是瓶颈
  改进计划：
    1. CR 时间窗口：每天 10:00-11:00、16:00-17:00 集中CR
    2. CR 轮值制度：每日1名CR值班人，保证24h内响应
    3. 大PR拆分指导：单PR控制在400行以内

Do（执行）—— 下一个Sprint执行
  - 第1周：试运行CR时间窗口
  - 第2周：加入CR轮值表
  - 持续记录PR合并时长数据

Check（检查）—— Sprint回顾会
  - PR合并时长从48h降至18h ✓
  - 但发现：CR值班人当天开发效率下降约30%
  - 需要调整：值班人当天减少50%的开发任务分配

Act（标准化/迭代）
  - 将CR值班制度写入团队规范
  - 调整值班人的Sprint容量评估
  - 下个PDCA循环：关注CR评论质量（是否只在挑格式问题）
```

**不同成熟度团队的度量策略：**

| 团队成熟度 | 度量策略 | 重点 | 避免 |
|------------|----------|------|------|
| 新建团队（0-6月） | 少量基础指标 | 交付节奏、基础质量 | 不要引入复杂指标，先稳定流程 |
| 成长团队（6-18月） | 逐步丰富维度 | 效率瓶颈、协作质量 | 不要急于横向对比个人 |
| 成熟团队（18月+） | 自驱度量 | 创新指标、技术债管理 | 避免指标僵化、过度优化 |

### ⚡ 实战经验

- **不要用代码行数度量生产力**：这是游戏行业最经典的反模式。一个优秀的优化可能删减 2000 行代码，效果远胜于新增 5000 行
- **游戏行业特殊指标**：帧率/内存/包体大小/加载时间等运行时性能指标也属于团队效能的一部分——技术指标和工程指标的交叉分析往往能发现深层问题（如内存泄漏高发期通常对应赶进度阶段）
- **警惕「仪表盘综合症」**：花两周搭一个华丽的数据看板，但从不基于数据采取行动。度量只是手段，改进才是目的。每周效能周报必须附带至少一个「下周改进项」
- **个人度量的陷阱**：永远不要公开排名个人生产力指标。一旦这么做，团队协作就会变成内部竞争，知识分享和互助行为会迅速消失
