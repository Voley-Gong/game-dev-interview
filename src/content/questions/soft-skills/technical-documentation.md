## 参考答案

### ✅ 核心要点

1. **文档分层**：决策文档、设计文档、API 文档、运维手册各有不同读者和更新频率
2. **文档即代码**：跟代码一起管理（Git/Wiki），版本对齐，PR 审查
3. **模板降低门槛**：提供标准模板，让写文档变成"填空"而非"写作文"
4. **嵌入开发流程**：设计评审必须先有文档，CR 检查文档更新
5. **定期保鲜**：设定过期机制，过时文档比没有文档更危险

### 📖 深度展开

**游戏客户端团队的文档分层体系：**

```
📚 文档体系架构

├── 📋 项目级文档（制作人/主程维护，季度更新）
│   ├── 技术架构总览.md        → 整体架构图 + 模块职责
│   ├── 技术选型决策记录.md     → ADR (Architecture Decision Records)
│   └── 项目里程碑与风险.md     → 当前状态 + 关键风险
│
├── 📐 模块级文档（模块负责人维护，迭代更新）
│   ├── 模块设计文档.md         → 架构、数据流、核心算法
│   ├── 接口文档.md             → 对外 API + 依赖关系
│   └── 性能基线.md             → 关键指标和压测数据
│
├── 🔧 操作级文档（全员贡献，随时更新）
│   ├── 开发环境搭建.md         → 新人第一天能跑起来
│   ├── 构建发布流程.md         → 如何出包、如何热更
│   ├── 常见问题排查.md         → FAQ + 已知问题
│   └── 编码规范.md             → 代码风格 + 命名约定
│
└── 📝 复盘级文档（迭代后更新）
    ├── 技术评审纪要.md         → 为什么选 A 不选 B
    ├── Bug 复盘.md             → 根因 + 改进措施
    └── 性能优化记录.md         → 优化前后数据对比
```

**ADR（架构决策记录）模板示例：**

```markdown
# ADR-012: 使用 BMFont 替代动态字体渲染

**状态：** 已采纳（2025-03-15）
**决策者：** 主程 + 引擎组
**背景：**
  中端 Android 设备 UI 渲染瓶颈，Profiler 显示动态字体打断合批，
  DrawCall 达 800+，帧率仅 22fps。

**决策：**
  使用 BMFont 位图字体方案，预渲染字符到纹理图集。

**方案对比：**
  | 方案 | 性能提升 | 实现成本 | 多语言支持 | 动态字号 |
  |------|---------|---------|-----------|---------|
  | BMFont | ⭐⭐⭐⭐⭐ | 中 | 需预生成 | 不支持 |
  | TextMeshPro | ⭐⭐⭐ | 低 | 好 | 支持 |
  | 保持现状 | ❌ | 无 | - | - |

**后果：**
  ✅ DrawCall 从 800 降至 120，帧率提升至 48fps
  ⚠️ 多语言版本需要预生成各语言字库，增加了构建流程复杂度
  📌 字号变更需要重新生成字体集，UI 系统需适配
```

**让团队持续写文档的 5 个机制：**

| 机制 | 具体做法 | 效果 |
|------|----------|------|
| 模板化 | 为每种文档类型提供 Markdown 模板，填空式写作 | 降低写作门槛 70% |
| 流程嵌入 | 设计评审前提 = 必须有设计文档；提测前 = 必须更新 API 文档 | 文档不再是"额外工作" |
| CR 检查 | Code Review 时检查"是否有文档变更" | 文档和代码同步更新 |
| 过期机制 | 文档头标注 `最后更新: YYYY-MM-DD`，超过 3 个月自动标记 ⚠️ | 防止过时文档误导 |
| 仪式感 | 每月"文档日"——半天时间专门整理文档 | 偿还文档债务 |

**不同阶段的文档策略：**

| 项目阶段 | 重点文档 | 容忍度 |
|----------|----------|--------|
| 原型期（0-3月） | 架构总览 + 环境搭建 | 允许粗粒度，快速迭代 |
| 正式开发（3-12月） | 模块设计 + 接口文档 + 规范 | 必须完整，CR 把关 |
| 上线运营（12月+） | 运维手册 + FAQ + 复盘 | 实时更新，定期校对 |

### ⚡ 实战经验

- **新人测试法**：每个新入职的成员就是文档质量的"活体测试"。他能不能仅靠文档在第一天跑通项目？第二天理解架构？如果不行，不是他的问题，是文档的问题
- **警惕"文档完美主义"**：有些团队追求文档详尽到极致，结果维护成本太高反而没人更新。好的文档是"足够用 + 能找到"，不是"百科全书"
- **工具选择要务实**：Confluence 适合大团队正式文档，Notion 适合中小团队快速迭代，Git Wiki 适合跟代码强关联的文档。游戏团队我推荐 Git Wiki + 代码内注释，降低工具切换成本
- **过时文档比没文档更危险**：一份描述旧接口的文档会让调用方浪费半天 Debug。强制要求文档头注明"适用版本"和"最后验证日期"，定期清理过时文档
