## 参考答案

### ✅ 核心要点

1. **会前预沟通**：关键干系人提前 1on1 对齐，避免会上意外反对
2. **结构化叙事**：问题定义 → 方案概览 → 关键决策点 → 风险与对策 → 资源需求
3. **数据驱动论据**：用 benchmark 数据、竞品案例、线上指标说话，而非"我觉得"
4. **预判反对意见**：列出 top 3 可能被质疑的点，主动回应而非被动挨打
5. **明确 Ask**：会议结束时让评审者清楚地知道你需要他们决策什么

### 📖 深度展开

**会前：90% 的成功发生在会议室之外**

很多技术人以为方案评审拼的是 PPT 和口才，实际上最关键的是会前预沟通：

```
评审前 3-5 天：
├── 发送一页纸摘要（Problem + Proposal + Impact）
├── 找核心决策者 1on1（15分钟即可）
│   ├── "关于下周的技术评审，想提前听听您的想法"
│   ├── 了解对方的顾虑和偏好
│   └── 如果对方有担忧，提前调整方案或准备数据
├── 找潜在反对者沟通
│   ├── "我知道您对 X 方案有顾虑，我想确认一下..."
│   └── 把反对意见变成方案改进的输入
└── 找中立派建立同盟
    └── "这个方案对你们组的影响是..."
```

**方案讲述的"金字塔结构"：**

| 层级 | 内容 | 时间占比 | 关键原则 |
|------|------|----------|----------|
| 顶层 | 核心结论（我们要做什么、为什么） | 10% | 30秒内说清楚 |
| 第二层 | 3个关键论据（为什么是这个方案） | 40% | 每个论据配数据 |
| 第三层 | 风险与对策 | 30% | 主动暴露风险 = 增加可信度 |
| 底层 | 资源需求与下一步 | 20% | 明确 Ask |

**对比：两种讲述方式的效果差异**

❌ **流水账式（工程师常见习惯）：**

> "首先我们调研了 A 方案，然后看了 B 方案，A 方案的优点是……缺点是……B 方案的优点是……缺点是……最后我们觉得 C 方案比较好……"
>
> → 评审者 5 分钟后开始看手机

✅ **结论先行式：**

> "我们建议采用 C 方案。三个理由：第一，性能提升 40%（展示 benchmark）；第二，团队已有经验，上手成本低；第三，社区活跃，长期维护风险小。当然，C 方案有一个风险是 X，我们的对策是 Y。今天需要大家确认的是 Z。"
>
> → 评审者立刻进入"验证模式"而非"理解模式"

**应对质疑的"接-转-连"技巧：**

```
接（Acknowledge）：承认对方的关注点
    "这个问题很关键，谢谢提出。"

转（Bridge）：引导到你准备好的数据
    "关于这一点，我们专门做了验证......"

连（Link）：回到你的主线
    "所以这也是为什么我们选择了 C 方案中的 C-2 子方案。"
```

**示例对话：**

> 评审者："你们这个方案迁移成本太高了吧？"
>
> ❌ "还好吧，我们估计就两周。"（防御性回应）
>
> ✅ "您对迁移成本的关注非常对。我们做了详细评估：迁移分 3 个阶段，每阶段 1 周，核心模块可回滚。第一阶段是非关键路径的 UI 层，风险最低。而且如果不迁移，下半年的维护成本会是迁移成本的 3 倍——这是我们整理的对比数据。"

### ⚡ 实战经验

- **游戏行业特殊场景**：策划突然在评审会上提出"那如果策划需求变了呢？"——预案中永远要有"灵活性评估"环节，说明方案的扩展点和弹性边界
- **避免"技术自嗨"**：评审会上制作人/发行方关心的是成本和风险，不是你的架构有多优雅。讲业务影响，不讲实现细节，除非被追问
- **准备"附录页"**：把深度技术细节放在附录，主线讲业务逻辑。被追问时翻到对应附录，展示你有准备但不浪费时间
- **会后 24 小时内发纪要**：包含决策结论、待办事项、负责人、截止日期。口头通过的方案没有书面记录等于没通过
