## 参考答案

### ✅ 核心要点

1. **先稳住自己**：你慌了团队就散了，给自己 15 分钟冷静分析再行动
2. **分类定级**：区分"真危机"和"感觉像危机"——评估实际影响范围和严重程度
3. **信息透明**：对上（管理层）和对下（团队）同步真实情况，不要藏雷
4. **分阶段应对**：止血（快速修复）→ 诊断（根因分析）→ 恢复（长期方案）→ 复盘（流程改进）
5. **关注团队状态**：危机中人的状态比技术方案更重要，注意团队成员的身心负荷

### 📖 深度展开

**场景一：线上严重事故（P0级）**

> 上线后第二天，大量玩家反馈存档丢失。影响范围约 15% 活跃用户，论坛和社群已经炸了，运营在催技术回复。

**危机应对时间线：**

```
T+0（事故爆发）
├── 确认事故真实性和影响范围（不要急着回应，先搞清楚事实）
├── 拉起紧急响应群（技术+运营+PR+制作人）
└── 指定一人对外统一发声（通常是你或制作人）

T+15min（止血阶段）
├── 判断能否快速回滚？→ 可以就立即回滚
├── 不能回滚则评估降级方案（关闭问题入口、发公告停服维护）
├── 分配任务：一人查日志定位、一人准备回滚/修复方案、你协调资源
└── 向上级汇报：现状 + 正在做什么 + 下一次更新时间

T+1h（诊断阶段）
├── 定位根因（数据库迁移脚本 bug 导致部分用户存档被覆盖）
├── 评估修复方案：热修 vs 停服维护 vs 版本回退
├── 准备用户补偿方案（与运营/策划协商）
└── 每小时向上级和运营同步进展

T+4h（恢复阶段）
├── 执行修复（从备份恢复存档 + 修复迁移脚本）
├── 灰度验证修复效果
├── 准备事故公告和补偿邮件
└── 安排轮班监控，不要让同一批人连轴转 24 小时

T+24h~48h（复盘阶段）
├── 组织事故复盘会（ blameless，对事不对人）
├── 输出事故报告（时间线、根因、改进措施）
├── 把改进项纳入迭代计划，设责任人
└── 关注团队情绪——经历过严重事故的团队需要恢复期
```

**场景二：核心技术人员突然离职**

> 你的主力客户端架构师（负责核心战斗系统）提出离职，2 周后走人。项目正处于关键开发期。

**应对策略：**

| 维度 | 行动 | 时间节点 |
|------|------|----------|
| 知识保全 | 立即安排交接：核心模块文档化、关键设计决策记录 | 第1周 |
| 风险评估 | 识别只有他掌握的"知识孤岛"，评估替代难度 | 第1-3天 |
| 团队稳心 | 1on1 安抚其他成员，防止连锁离职 | 第1周 |
| 人员补充 | 评估内部能否消化，还是需要紧急招聘/外包 | 第1周 |
| 项目调整 | 与制作人沟通是否需要调整里程碑 | 第1周 |

**关键对话示例——与离职人员的沟通：**

```
"我尊重你的决定。在我心里你是团队的核心，但我不想用感情绑架你。
现在最重要的是做好交接，让项目和你都能平稳过渡。

我想了解几点：
1. 是什么让你决定离开？（真实原因，改进我的管理）
2. 你手上哪些工作是别人不清楚的？我们列个清单。
3. 交接期间你希望怎么安排？我会尽量配合你的节奏。

不管怎样，以后有需要帮忙的，随时找我。"
```

**场景三：项目严重延期，士气低落**

> 项目原定 3 个月交付，已经第 4 个月了，还有 30% 功能未完成。团队连续加班一个月，疲惫不堪，有人开始消极怠工。

**危机领导力三步走：**

**第一步：重新校准现实**

```
不要继续用原来的计划自欺欺人。

召开核心团队会议：
"我知道大家都很累。现在我们需要诚实地面对现状。
我请大家做一件事：忘掉原来的计划，基于当前进度，
给我一个你们认为现实的交付时间。"

→ 重新评估：
  - Must have vs Nice to have 拆分
  - 可砍掉的功能列表
  - 真实的剩余工作量估算
```

**第二步：争取资源和调整预期**

```
拿着重新评估的结果去找制作人/发行方：

"现在的进度如下：
A方案：砍掉 P3 功能，4 周后交付核心版本
B方案：保留全部功能，需要 8 周再加 2 名开发
C方案：分阶段交付，先上核心版，后续补内容

我推荐 A 方案。需要你的决策。"

→ 不要只报问题不给方案
→ 让决策方做选择题，而非问答题
```

**第三步：重建团队士气**

```
具体行动：
1. 停止无效加班——加班已经一个月了还在延期，说明加班不能解决问题
2. 给团队放 1-2 天调休假——休息后效率远高于疲劳硬撑
3. 调整目标后正式跟团队同步——"新计划是这样的，我们有信心按这个节奏完成"
4. 在小里程碑达成时庆祝——哪怕只是订个下午茶
5. 你自己也要展现出信心——团队会读取你的情绪状态
```

### ⚡ 实战经验

- **危机中的沟通频率要翻倍**：平时一周一次的汇报，危机时一天一次甚至半天一次。上级最怕的不是出事，是出事了还不知道
- **Blameless 文化**：事故复盘绝不追究个人责任。追究责任只会让下次事故被隐瞒得更久
- **游戏行业特殊场景——上线首周是高危期**：提前安排好值班表、回滚预案、应急联系人。不要上线当天才发现没做压力测试
- **记住你自己也是人**：管理者在危机中容易忽视自己的状态。你连续奋战 48 小时后的决策质量可能比休息后的决策差 50%。学会授权后去休息
- **预防是最好的危机管理**：建立监控告警、代码灰度发布、自动化测试、核心人员 backup 计划。平时多流汗，战时少流血
