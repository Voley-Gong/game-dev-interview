## 参考答案

### ✅ 核心要点

**STAR 法则：**
- **S**ituation（情境）：背景和挑战
- **T**ask（任务）：你的职责和目标
- **A**ction（行动）：你具体做了什么
- **R**esult（结果）：量化的成果

### 📖 深度展开

**示例 1：「描述一次你解决技术难题的经历」**

> **S**：我们的卡牌游戏在中端 Android 设备上帧率只有 22fps，大量玩家投诉卡顿，评分从 4.6 降到 3.8。
>
> **T**：作为客户端主程，我需要在两周内将中端机帧率提升到稳定 40fps 以上。
>
> **A**：
> 1. 使用 Profiler 定位瓶颈，发现 UI 渲染占了 60% 帧时间
> 2. 分析发现是大量 Label 组件打断合批，DrawCall 高达 800+
> 3. 设计了 BMFont 位图字体方案替代动态文本渲染
> 4. 重构 UI 层级，按组件类型分组排列
> 5. 建立性能基准测试流程，每周自动回归
>
> **R**：中端机帧率从 22fps 提升到 48fps，DrawCall 从 800 降至 120。评分恢复到 4.5，当月 DAU 增长 15%。

**示例 2：「描述一次你和团队成员意见分歧的经历」**

> **S**：新项目的技术选型会议上，团队对使用 Cocos 还是 Unity 产生激烈分歧。
>
> **T**：作为技术负责人，我需要在一周内做出决策，并让团队接受。
>
> **A**：
> 1. 不急于拍板，先让双方各自准备技术评估报告
> 2. 组织一场 2 小时的技术评审会，从 5 个维度打分（团队能力、项目需求、生态支持、性能表现、长期维护）
> 3. 我自己准备了一份风险评估矩阵
> 4. 最终发现项目需求（2D 为主 + 小游戏出海）更匹配 Cocos，而团队经验也偏向 Cocos
>
> **R**：团队达成共识选了 Cocos。后续项目进展顺利，3 个月完成 MVP。Unity 派的成员也认可了决策过程。

**常见行为面试问题清单：**

| 问题类型 | 示例问题 |
|----------|----------|
| 技术攻坚 | 描述一次你解决过的最难的技术问题 |
| 领导力 | 描述一次你推动团队改进的经历 |
| 冲突处理 | 描述一次你与同事意见不一致的经历 |
| 失败经验 | 描述一次失败的项目经历，你学到了什么 |
| 时间管理 | 描述一次 deadline 压力下你如何优先排序 |

### ⚡ 实战经验

- **提前准备 3 个故事**：技术攻坚 / 团队管理 / 失败复盘
- **结果必须量化**：「提升了性能」不如「帧率从 22 提升到 48fps」
- **控制在 2-3 分钟**：不要超过 3 分钟，否则面试官会走神
- **诚实**：不要编造，面试官会追问细节
- **「我」而非「我们」**：说清你个人的贡献，不要全说「团队做了XX」
