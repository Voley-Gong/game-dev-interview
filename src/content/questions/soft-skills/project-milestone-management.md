## 参考答案

### ✅ 核心要点

1. **里程碑拆分**：将大目标拆为可验证的阶段性交付物（Vertical Slice 思维）
2. **任务粒度控制**：单个任务不超过 3 人天，超出则继续拆分
3. **每日同步与风险预警**：站会不只是报流水账，重点是暴露阻塞
4. **缓冲区设计**：每个里程碑预留 15-20% 的缓冲时间应对突发
5. **里程碑回顾**：每个节点做 Demo 评审 + 复盘，而非等到项目结束

### 📖 深度展开

**里程碑拆分策略（以一款 RPG 手游为例）：**

| 里程碑 | 时间跨度 | 核心交付物 | 验收标准 |
|--------|----------|------------|----------|
| M1 - 核心战斗 Prototype | 4 周 | 战斗系统可玩原型 | 核心战斗 loop 跑通，内部试玩通过 |
| M2 - Vertical Slice | 8 周 | 15 分钟完整体验 | 包含战斗+UI+存档+一段关卡，可对外演示 |
| M3 - Alpha | 12 周 | 全功能但不打磨 | 所有系统接入，主流程可跑通 |
| M4 - Beta | 8 周 | 内容填充 + 优化 | 性能达标，Bug 收敛到可接受范围 |
| M5 - 发布候选 | 4 周 | 上线版本 | 通过平台审核，0 P0 Bug |

**每日站会的正确打开方式：**

```
❌ 错误示范（流水账式）：
"昨天写了登录模块，今天继续写注册模块，没有问题。"

✅ 正确示范（信息密度高）：
"昨天完成了登录模块的接入，但发现服务器返回格式和文档不一致，
已经找服务端同学确认，预计今天下午能修复。这个可能导致注册模块
延后半天，我会在下午同步进度。"
```

站会三问的真正目的：
- **昨天做了什么** → 让团队知道进度偏差
- **今天做什么** → 让协作方知道你的工作与他们的依赖关系
- **有什么阻塞** → 尽早暴露风险，而不是自己闷头解决

**进度风险的识别与应对：**

```
风险信号（需要主程/PM 警觉）：

🚨 红色信号（立即介入）：
   - 某个任务连续 3 天没有进展
   - 开发者频繁加班但仍未完成
   - 某个系统集成时发现接口完全对不上
   - 测试返工率超过 30%

⚠️ 黄色信号（持续关注）：
   - 任务实际耗时持续超过估算的 1.5 倍
   - 需求在开发中途频繁变更
   - 团队成员对某个模块的实现方式存在分歧
   - 跨部门协作出现等待超过 2 天

应对策略：
1. 红色 → 当天介入，要么加人、要么砍需求、要么换方案
2. 黄色 → 在下次里程碑评审中正式提出，制定缓解计划
```

**燃尽图的正确解读：**

```
理想燃尽线是一条直线，但实际上：

    任务量
     |
  100 |*                          
     |  *    ← 前期进度慢正常（搭框架、铺基础设施）
   75 |    *                      
     |      *    ← 中期应该加速
   50 |        *                  
     |          **    ← 后期减速也要控住（打磨和修 Bug）
   25 |             *
     |               *
    0 +---|----|----|----|----→ 时间
       M1   M2   M3   M4   M5

如果中期没有加速 → 说明有系统性阻塞
如果后期断崖式下降 → 说明任务拆分不够细
如果全程匀速 → 可能有人在虚报进度
```

### ⚡ 实战经验

- **Vertical Slice 是最重要的里程碑**：它验证了"全流程能跑通"，比"80% 的模块各自完成了"有价值得多。很多项目死在各模块各自为战、无法集成
- **永远不要相信"90% 完成了"**：游戏开发中最后的 10% 往往需要 50% 的时间。打磨、性能优化、兼容性适配这些"收尾"工作是最容易超时的
- **里程碑评审要有 Demo**：PPT 讲十页不如让大家玩 5 分钟。Playable Demo 是最真实的进度验证
- **需求变更要走流程**：策划中途改需求，不能直接口头说。要走变更评审，评估影响范围、调整排期，让制作人签字。这不是官僚主义，是保护团队
- **Buffer 不要明着说**：你心里留 20% 缓冲，但对外不要说"我留了 buffer"，否则上级会认为你的估算不准。直接把 buffer 摊进每个任务的工时里
