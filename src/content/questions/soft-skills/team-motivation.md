## 参考答案

### ✅ 核心要点

1. **理解个体驱动力**：每个人在乎的东西不同（成长、认可、自主性、安全感）
2. **意义感建设**：让成员理解自己的工作如何影响产品和用户
3. **合理授权**：给挑战而非 micromanage，信任是最好的激励
4. **即时认可**：具体、及时的正向反馈比年度考核更有效
5. **处理低士气**：先诊断根因（加班疲劳？方向迷茫？人际矛盾？）再对症下药

### 📖 深度展开

**开发者驱动力模型（基于 Self-Determination Theory 适配游戏行业）：**

```
                    技术成长
                       ↑
                       │
       认可与尊重 ←──┼──→ 自主与掌控
                       │
                       ↓
                    意义与影响

每个成员的驱动力组合不同：

  资深渲染程序员    → 自主性 40% + 技术成长 35% + 认可 15% + 意义 10%
  初中级客户端开发  → 技术成长 45% + 认可 25% + 自主性 20% + 意义 10%
  引擎/工具开发    → 自主性 35% + 意义 25% + 技术成长 25% + 认可 15%
```

**不同场景的激励策略：**

| 场景 | 症状 | 策略 |
|------|------|------|
| 长期加班期 | 效率下降、沉默、抱怨增多 | 明确截止时间、调休承诺兑现、引入外部新鲜刺激（技术分享） |
| 项目方向频繁变更 | 无力感、"反正还会改"心态 | 让团队参与方向决策、解释变更原因、保护核心开发节奏 |
| 技术栈老旧 | 成员担心竞争力 | 安排技术探索时间（20% 规则）、支持开源贡献、引入新技术试点 |
| 核心成员想离职 | 影响团队信心 | 1on1 深谈了解真实原因、真诚挽留但不画饼、做好知识传承 |
| 发版后疲惫期 | 产出下降、松懈 | 主动安排恢复期、项目复盘庆功、让成员选下一个感兴趣的模块 |

**授权的四个层级：**

```
层级 1：指令式（我决定，你执行）
  → 适用于：紧急情况、新人培训初期

层级 2：建议式（我给方向，你来执行）
  → 适用于：中级开发者的常规任务

层级 3：协商式（你出方案，我们一起拍板）
  → 适用于：高级开发者的技术方案

层级 4：委托式（你全权负责，定期同步结果）
  → 适用于：核心骨干的独立模块

主程常见误区：对所有人和所有事都用层级 1 或层级 2。
正确做法：根据人的能力和事的确定性灵活切换。
```

**正向反馈的具体化对比：**

❌ **笼统型（无效）：**
> "最近干得不错，继续加油！"

✅ **具体型（有效）：**
> "你上周主动重构了 UI 模块的合批逻辑，DrawCall 从 300 降到 80。这个优化直接改善了中端机的帧率表现，玩家社区已经有正面反馈了。做得很好。"

**低士气诊断清单：**

```
当你感觉团队士气低落时，逐项排查：

□ 工作量：是否持续超负荷？连续几周？
□ 清晰度：团队是否清楚项目目标和自己的角色？
□ 公平感：有人觉得分配不公或评价偏颇吗？
□ 成长感：成员觉得在过去 3 个月有进步吗？
□ 掌控感：成员对自己的工作有决策权吗？
□ 人际关系：团队内有没有未解决的矛盾？
□ 向上信心：团队对项目管理层（制作人/总监）有信任吗？
```

### ⚡ 实战经验

- **画饼是最差激励**：游戏行业太多"成功后分红"的空头支票。不如用可见的短期激励——让成员在技术分享会上展示成果、给一个有挑战的独立模块、支持他选一个感兴趣的技术方向深入研究
- **复盘会不是批判会**：项目复盘要遵循"对事不对人"原则。问"流程哪里出了问题"而不是"谁的错"。好的复盘会让团队感到安全，差的复盘会摧毁信任
- **技术分享是双向激励**：让成员做技术分享，表面上是知识传播，实际是给分享者舞台和认可。听众也能感受到团队的技术氛围在提升
- **离职是最好的反馈**：当有人离职时，认真做 exit interview。他们说的真话比在职的人多 10 倍。把这些反馈变成团队改进的输入
