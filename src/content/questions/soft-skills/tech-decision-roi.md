## 参考答案

### ✅ 核心要点

1. **明确决策语境**：先问"解决什么问题"，再问"用什么方案"
2. **多维度评估矩阵**：不只看技术先进性，还要看团队、成本、时间、风险
3. **ROI 思维**：投入产出比不只是钱，还包括学习成本、维护成本、机会成本
4. **决策记录与回顾**：写下 why，定期复盘决策是否正确
5. **避免过度工程**：用最简方案解决当前问题，为未来留扩展性但不提前实现

### 📖 深度展开

**技术选型评估矩阵（实战模板）：**

```
评分维度（每项 1-5 分）：

| 维度           | 权重 | 方案 A | 方案 B | 方案 C |
|---------------|------|--------|--------|--------|
| 团队匹配度     | 25%  |        |        |        |
| 开发效率       | 20%  |        |        |        |
| 性能表现       | 15%  |        |        |        |
| 生态与社区     | 10%  |        |        |        |
| 学习成本       | 10%  |        |        |        |
| 长期维护性     | 10%  |        |        |        |
| 招聘难度       | 5%   |        |        |        |
| 风险（负向）   | 5%   |        |        |        |
| 加权总分      | 100% |        |        |        |
```

**经典案例：自研引擎 vs 商业引擎**

> **场景**：一款中型体量的卡牌+SLG 手游，团队 15 人，预算周期 18 个月。
>
> | 考量 | 自研引擎 | Cocos Creator | Unity |
> |------|---------|---------------|-------|
> | 开发效率 | 低（要先造轮子） | 高（生态成熟） | 高（生态最成熟） |
> | 团队匹配 | 低（无经验） | 中（部分经验） | 高（全员经验） |
> | 性能控制 | 极高（完全可控） | 中（够用） | 中高（可控但复杂） |
> | 长期成本 | 极高（持续投入） | 低（免费/分成） | 中（授权费） |
> | 招聘难度 | 极高 | 中 | 低 |
>
> **决策**：选择 Cocos Creator。
>
> **关键洞察**：自研引擎的性能优势在这个项目体量下根本体现不出来，而 18 个月的 deadline 意味着开发效率是第一优先级。一个"技术不够完美但按时上线"的游戏，远胜于一个"技术完美但胎死腹中"的游戏。

**ROI 思维的三层模型：**

```
第一层：直接成本
  ├── 开发工时（人月）
  ├── 工具/引擎授权费
  └── 服务器/硬件成本

第二层：隐性成本
  ├── 学习曲线（团队多久能上手）
  ├── 维护成本（谁来做日常维护）
  ├── 技术债务（未来要还的代价）
  └── 机会成本（选了这个就不能选那个）

第三层：战略价值
  ├── 团队能力沉淀（是否可复用）
  ├── 技术品牌（是否有助于招聘/融资）
  └── 长期演进空间（3年后还能用吗）
```

**Tradeoff 决策清单：**

当你面临"既要又要"的时候，用以下问题强制排序：

1. **如果不做这个技术决策，会怎样？**（评估不作为的代价）
2. **半年后这个决策还重要吗？**（区分临时和长期问题）
3. **最坏情况是什么？我们能承受吗？**（风险评估）
4. **团队是否认同？**（执行落地的前提）
5. **有没有中间方案？**（避免非黑即白的思维）

**过度工程的识别信号：**

| 信号 | 说明 |
|------|------|
| "万一以后需要…" | YAGNI——You Aren't Gonna Need It |
| 团队没人能看懂 | 复杂度超出了团队认知负荷 |
| 花了一周设计，一天实现 | 设计/实现比严重失衡 |
| 解决的是假想问题 | 没有真实场景驱动 |

### ⚡ 实战经验

- **POC 验证比 PPT 争论有效 100 倍**：两个方案争执不下时，各花 2 天做 spike，用数据说话
- **"团队匹配度"权重应该最高**：一个团队不熟悉的技术栈，再先进也是灾难。技术可以学，但 deadline 不会等
- **记录决策的 why**：半年后你一定会忘记当初为什么选 A 不选 B，写一份 ADR（Architecture Decision Record），未来感谢自己
- **定期复盘**：每个季度回顾一次重大技术决策。"当时选 Cocos 是对的吗？如果重来会怎么选？"——这种复盘是团队成长的加速器
