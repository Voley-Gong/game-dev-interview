## 参考答案

### ✅ 核心要点

1. **多层考核维度**：不能只看代码产出，要覆盖代码质量、技术深度、协作贡献、业务理解四个维度
2. **过程 + 结果双轨**：OKR 管目标结果，360 评估管协作过程，技术雷达管能力成长
3. **定期 + 实时结合**：季度正式 Review + 每双周 1on1 实时反馈，避免「年底惊喜」
4. **区分等级与淘汰机制**：清晰的晋升通道和不匹配处理流程，考核才有约束力
5. **数据驱动但拒绝纯量化**：代码行数 / Bug 数等单一指标会扭曲行为，必须结合定性判断

### 📖 深度展开

**技术团队绩效考核维度矩阵：**

| 维度 | 权重 | 评估方式 | 关键信号 |
|------|------|----------|----------|
| 代码质量 | 25% | CR 通过率 + 架构评审 + 线上事故率 | 是否主动重构、是否写测试、CR 评论质量 |
| 技术深度 | 25% | 技术分享 + 难题攻坚记录 + 技术文档 | 是否能独立解决 P0 级技术难题、是否有技术输出 |
| 协作贡献 | 25% | 360 评估 + 跨部门反馈 | 是否帮新人成长、是否主动补位、CR 是否认真 |
| 业务理解 | 25% | 需求评审参与度 + 方案业务合理性 | 是否能从业务角度提建议、是否理解优先级 |

> **关键原则**：权重不是固定的。初级偏代码质量（40%），高级偏业务理解和协作（各 35%），因为高级工程师的核心价值是放大团队产出。

**OKR 设定示例（客户端主程团队）：**

```
O：提升游戏客户端质量和开发效率

  KR1：线上 Crash 率从 0.8% 降至 0.3%
  KR2：包大小从 850MB 优化至 600MB 以内
  KR3：热更新流程从 4 小时缩短至 30 分钟
  KR4：团队中至少 2 人能独立负责完整模块开发
```

**360 评估问卷示例（节选）：**

```
协作维度（同事互评，1-5 分）：

1. 与此人合作完成任务的体验如何？
2. 此人在技术讨论中是否能理性沟通、尊重不同意见？
3. 此人是否主动帮助你或团队解决了问题？
4. 此人的 CR 反馈是否有建设性、帮助你成长？
5. 如果组建新团队，你是否愿意再次与此人合作？

开放问题：
- 他/她最让你欣赏的一点是什么？
- 你觉得他/她最需要改进的地方是什么？
```

**绩效面谈框架（1on1 季度 Review）：**

```
时间：45-60 分钟，提前 3 天通知对方准备自评

流程：
1. 自评（10 min）：让对方先说——本期亮点、不足、目标完成情况
2. 反馈（15 min）：你的观察——认可亮点 + 指出问题（用具体事例）
3. 对齐（10 min）：双方对考核结果达成共识（如有分歧，留记录）
4. 规划（15 min）：下季度目标 + 成长方向 + 需要的支持
5. 记录（10 min）：书面记录双方确认，HR 系统

话术示例：
✅ "这个季度你在战斗模块的性能优化上做了很好的工作，
    尤其是 DrawCall 优化方案，降低了 40%。"
✅ "我注意到最近两次 CR 你只回复了 'LGTM'，
    我希望你能给出更具体的反馈，这对团队知识传递很重要。"
❌ "你的考核结果是 B"（不要直接甩结果，要先讨论过程）
```

**不同绩效等级的处理方式：**

| 等级 | 占比建议 | 处理方式 |
|------|----------|----------|
| S（卓越） | 5-10% | 重点培养，给予更多技术决策权，可选晋升 |
| A（优秀） | 20-30% | 正向激励，分配核心模块 |
| B（达标） | 50-60% | 维持现状，明确提升方向 |
| C（待改进） | 10-15% | 制定 PIP（绩效改进计划），8 周观察期 |
| D（不匹配） | <5% | 启动转岗或离职流程 |

### ⚡ 实战经验

- **警惕古德哈特定律**：「当一个指标变成目标时，它就不再是好指标。」代码行数、Bug 修复数绝不能直接当 KPI，否则工程师会刷量
- **游戏行业特殊性**：项目周期长（1-3年），不能用互联网的快速迭代标准考核。里程碑节点的交付质量比每周产出更重要
- **避免「轮流坐庄」**：S 和 A 不能轮流给，否则考核失去意义。宁可团队整体评分偏低，也不能为了「和谐」虚高
- **PIP 不是惩罚是机会**：给 C 等员工明确的时间线和可执行目标，帮他成功才是目的。但到期不达标，必须果断处理
- **新人保护期**：入职 3 个月内不参与正式考核，只做观察反馈，避免新人因环境不熟被打低分
