## 参考答案

### ✅ 核心要点

1. **理解本质**：敏捷不是某个框架，而是一组价值观——快速反馈、拥抱变化、持续交付
2. **因地制宜**：游戏开发有其特殊性（创意密集、质量门槛高、内容制作周期长），不能照搬软件行业的 Scrum
3. **选择框架**：Scrum 适合功能开发期，Kanban 适合内容持续产出期和上线运维期
4. **关键实践**：Sprint 规划、每日站会、迭代回顾、可演示交付
5. **度量效果**：用速度图（Velocity）、燃尽图（Burndown）和团队满意度三方交叉验证

### 📖 深度展开

**游戏项目中 Scrum vs Kanban 的实际选择：**

| 维度 | Scrum | Kanban |
|------|-------|--------|
| 节奏 | 固定 Sprint（通常 2 周） | 持续流动，无固定迭代 |
| 角色 | PO / SM / Dev Team | 无强制角色 |
| 适合阶段 | 核心玩法开发、系统功能开发 | 美术资源产出、运营活动、Bug 修复 |
| 计划变更代价 | Sprint 内冻结，下个 Sprint 调整 | 随时调整优先级 |
| 度量 | Velocity、Sprint Goal 达成率 | Lead Time、WIP Limit |
| 游戏行业适配度 | ★★★★ 核心开发期 | ★★★★ 内容/运维期 |

**实际落地的 Sprint 节奏（2周迭代）：**

```
第 1 天（周一）：Sprint Planning（4小时）
  - PO 拉需求池 Top 项，团队估点
  - 确认 Sprint Goal（一句话："这个 Sprint 交付可联机的战斗 Demo"）
  - 拆任务到 1 天以内粒度

每日：Daily Standup（15分钟，别超）
  - 只说三件事：昨天做了什么 / 今天做什么 / 有什么阻塞
  - 有讨论的话会后拉人，不要全员站着听

第 1 周周五：Mid-Sprint Review（1小时，可选）
  - 给 PO 看半成品，确认方向没错
  - 游戏开发特别重要——玩起来是什么手感，要早验证

第 2 周周四：Sprint Review + 可玩版本演示（2小时）
  - 实机操作演示，不是念 PPT
  - PO、制作人、甚至发行方参与体验

第 2 周周五：Retrospective（1.5小时）
  - 做得好的 / 需要改进的 / 下个 Sprint 的 Action Item
  - Action Item 必须 SMART，且不超过 3 个
```

**游戏行业特有的敏捷陷阱：**

```
陷阱 1：美术任务无法拆成小粒度
  ❌ "角色模型制作" 作为一个 Story，估 21 点
  ✅ 拆成：原画确认 → 低模 → 高模 → 拓扑 → UV → 贴图 → 骨骼绑定
     每个环节独立估点，流转可见

陷阱 2：Build 不可玩
  ❌ Sprint 末尾合版本，各种 Crash，没法演示
  ✅ 每日自动出包（CI/CD），保持 Master 永远可运行
     美术资源用 Asset Bundle 热更，不影响主干

陷阱 3：策划需求频繁变更
  ❌ PO 随时插需求，Sprint Goal 形同虚设
  ✅ Sprint 内冻结需求（紧急Bug除外）
     变更需求走变更流程，需制作人审批 + 等价需求移出
```

**敏捷度量指标设计：**

```
健康度仪表盘（每周更新）：

┌─────────────────────────────────────────┐
│  指标               │ 目标值  │ 当前值  │
├─────────────────────────────────────────┤
│  Velocity 趋势      │ 稳定±15%│         │
│  Sprint Goal 达成率 │ ≥ 80%  │         │
│  Bug 收敛率         │ 上升   │         │
│  Build 稳定性       │ ≥ 95%  │         │
│  团队满意度(1-5)    │ ≥ 3.5  │         │
│  加班时长           │ ≤ 10h  │         │
└─────────────────────────────────────────┘

注意：Velocity 是度量工具，不是考核工具！
一旦用来考核，团队就会学会 inflated estimation。
```

### ⚡ 实战经验

- **别教条**：面试中遇到说「我们严格按照 Scrum Guide」的候选人，大概率没真正做过。好的回答是「我们根据项目实际情况做了裁剪」
- **Sprint 长度别太长**：游戏行业 1 周太赶、3 周太松，2 周是甜区。如果团队不满 5 人可以考虑 1 周迭代
- **回顾会是最有价值的会议**：很多团队的 Retro 流于形式（写便签→念一遍→扔了）。关键是 Action Item 必须落实到人，下个 Sprint 检查执行情况
- **Demo 驱动开发**：游戏是体验型产品，每个 Sprint 必须有可玩版本。文字描述和实际手感之间有巨大鸿沟，早玩早纠偏
- **与技术主程的关系**：主程关注技术质量，Scrum Master 关注流程健康。如果你同时兼任，记得在两个角色之间切换思维——别用管理帽子做技术决策，也别用技术帽子管流程
