## 参考答案

### ✅ 核心要点

1. **建立规范**：代码规范、分支策略、CR 流程
2. **人才培养**：技术分享、1on1、成长路径
3. **流程建设**：自动化构建、CI/CD、质量门禁
4. **沟通协作**：跨部门对齐、向上管理、冲突调解
5. **技术决策**：选型评估、方案评审、债务管理

### 📖 深度展开

**代码规范与质量保障：**

```
推行步骤（渐进式，不要一步到位）：

第1周：引入 ESLint/Prettier（自动格式化，零争议）
第2周：定义核心编码规范（命名、模块结构）
第3周：引入 CR 流程（先只看架构，不纠结格式）
第4周：建立 CI 自动检查（格式 + lint + 单元测试）
第5周+：逐步提高 CR 标准和测试覆盖率
```

**1on1 沟通框架：**

```
每两周一次，30-45分钟

固定议程：
1. 近期工作感受（1-5分）
2. 遇到的困难/阻塞
3. 成长目标进展
4. 需要我提供什么支持
5. 对团队/流程的改进建议

关键原则：
- 多听少说（80/20）
- 不评价，只引导
- 记录并跟进
```

**技术决策流程：**

```
收到技术选型需求
  ↓
收集信息（团队调研 + 业界案例 + POC 验证）
  ↓
多维度评分矩阵
  ├── 团队能力匹配度
  ├── 项目需求契合度
  ├── 生态与社区活跃度
  ├── 性能天花板
  ├── 维护成本（长期）
  └── 风险评估
  ↓
技术评审会（核心成员参与）
  ↓
形成决策文档（记录 why，而非仅 what）
  ↓
定期回顾（3个月后评估决策是否正确）
```

### ⚡ 实战经验

- **不要技术独裁**：主程不是所有技术决策都一个人做，要尊重团队意见
- **CR 是教学工具**：不只是找 bug，更是知识传递和标准对齐
- **处理分歧**：数据说话，不主观判断。「A 和 B 哪个性能好？我们跑个 benchmark」
- **技术债务管理**：每个迭代留 20% 时间还技术债，否则雪球越滚越大
- **向上管理**：主动向制作人/总监汇报风险，不要等到出问题才说
- **保护团队**：帮团队挡住不合理的需求变更，维护开发节奏
