## 参考答案

### ✅ 核心要点

1. **对事不对人**：评论代码，不评论写代码的人
2. **区分优先级**：`[Blocker]` / `[Suggestion]` / `[Nit]` 三级标注，让作者知道哪些必须改
3. **提供方案而非仅指出问题**："建议用 XX 模式" 比 "这里写法不对" 有效 10 倍
4. **解释 Why**：不只说改什么，更说为什么改——这是知识传递的关键
5. **肯定好的设计**：CR 不只挑毛病，看到优雅的方案也要点名表扬

### 📖 深度展开

**评论分级体系（推荐在团队中推行）：**

| 标签 | 含义 | 处理方式 | 示例 |
|------|------|----------|------|
| `[Blocker]` | 必须修改才能合并 | 作者必须修改或给出合理解释 | `[Blocker] 这里空指针没判空，线上会崩` |
| `[Suggestion]` | 建议修改但不阻塞 | 作者酌情采纳，可以讨论 | `[Suggestion] 可以考虑用对象池优化，当前每帧 new 了 50 个对象` |
| `[Nit]` | 细节问题，可选 | 改了更好，不改也行 | `[Nit] 变量名 `tmpData` 改成更具体的名字更好` |
| `[Question]` | 纯粹请教 | 作者回答后决定是否修改 | `[Question] 这个排序逻辑为什么按等级而不是按时间？` |
| `[Praise]` | 点赞 | 不需要回复 | `[Praise] 这个状态机设计很清晰，学到了` |

**语气对比——同一问题的不同表达：**

```
❌ "这个写法是错的"
✅ "这里如果 player 为 null 会抛异常，建议加一层判空"

❌ "你为什么不抽成公共方法？重复代码太多了"
✅ "[Suggestion] 这段逻辑和 BattleManager 里的伤害计算很像，
   可以考虑抽成 DamageUtil.calculate()，后续维护更方便"

❌ "性能太差了，这样能上线？"
✅ "[Blocker] 这个 foreach 里每帧都 Instantiate，
   Profiler 显示 GC 每秒 2MB。建议改用对象池复用"

❌ "命名不规范"
✅ "[Nit] 团队规范里方法名用驼峰，`Get_PlayerData` → `getPlayerData`"
```

**CR 评论的"三明治"结构：**

```
┌─────────────────────────────────┐
│  正面反馈（表扬亮点）             │  ← 建立安全感
├─────────────────────────────────┤
│  核心问题（Blocker + Suggestion）│  ← 建设性建议
├─────────────────────────────────┤
│  鼓励 + 期待                     │  ← 收尾积极
└─────────────────────────────────┘

示例：
"整体架构很清晰，事件系统解耦得不错 👍
 [Blocker] 第 87 行资源加载没有走 AssetManager，会导致内存泄漏
 [Suggestion] 第 120 行的魔法数字可以提成常量
 期待下一版，继续加油！"
```

**不同资历的 CR 策略差异：**

| 作者资历 | CR 重点 | 语气策略 |
|----------|---------|----------|
| 新人（< 6 月） | 基础规范、命名、架构理解 | 多解释 Why，多鼓励，少 Nit |
| 中级（1-3 年） | 设计模式、性能、边界情况 | 平等讨论，鼓励提出替代方案 |
| 高级（3 年+） | 架构决策、tradeoff、技术债 | 尊重专业判断，聚焦分歧点深入讨论 |

### ⚡ 实战经验

- **游戏行业特殊性**：热更新代码的 CR 要格外严格——线上出 bug 无法直接发版修复，必须走热更流程。在 CR 中标注 `[Hotfix-critical]` 提醒作者这段代码的容错要求
- **异步 CR 规范**：游戏项目 crunch 期大家都很忙，约定 CR 响应时间（如 24 小时内首次回复），避免 PR 积压。可以用 Discord/飞书机器人自动提醒
- **不要在 CR 里讨论大方向**：如果发现架构层面有问题，约面对面或语音讨论，不要在评论里写一篇论文。CR 适合讨论"怎么实现"，不适合讨论"要不要这么做"
- **建立 CR 文化比规则更重要**：主程/技术负责人自己的 PR 也要接受 CR，而且要示范如何优雅地接受建议——这比任何规则文档都有效
