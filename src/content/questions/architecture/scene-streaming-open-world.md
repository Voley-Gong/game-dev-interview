## 参考答案

### ✅ 核心要点

1. **核心矛盾：世界无限大 vs 内存有限**。开放世界（GTA、塞尔达、原神）是一个连续的大地图，无法一次性载入内存（几十 GB）。解决方案是**流式加载（Streaming）**：把世界切成网格化的小块（Chunk/Cell/Tile），只保留玩家周围「可见范围」内的块常驻内存，玩家移动时动态地加载前方块、卸载后方块。整个过程对玩家透明——没有加载界面，世界像水一样「流」过来。

2. **与离散场景切换是两种对立范式**：`scene-management` 解决的是「A 场景 → 加载界面 → B 场景」的离散切换（有 loading screen、整场景原子加载）。流式加载则是「始终在一个世界，边界处无缝拼接」——没有 loading screen，块是增量的、按需的、连续的。原神能在提瓦特大陆任意奔跑不读条，靠的就是流式加载而非场景切换；从大世界进秘境那一刻的 loading 则是离散切换。

3. **网格化分块（Chunk Grid）是数据组织基础**：把世界按固定边长（如 256m × 256m）切成 2D 网格，每块包含自己的地形、物体、碰撞、NavMesh、LOD。分块大小是关键权衡——太大则加载一次耗时久、内存占用大；太小则块数爆炸、加载调度频繁。典型边长 128-512m，配合 LOD 让远景块只存低精度数据。

4. **加载/卸载的触发：玩家位置 + 预加载缓冲圈**。维护一个以玩家为中心的「加载圈」——内圈（紧邻玩家）必须常驻；外圈（缓冲带）提前异步加载，保证玩家走过去时块已就绪；圈外块进入「待卸载」队列。预加载距离要 > 玩家最大移动速度 × 单块加载耗时，否则会「走到没加载完的虚空」。卸载用 LRU + 引用计数，避免还有实体引用时被释放。

5. **帧预算 + 后台线程 = 无卡顿的关键**。加载绝不能在主线程同步做（一次同步 IO 直接卡几百 ms）。正确做法：①**异步 IO**（后台线程读磁盘/解压）②**分帧提交**——每帧只把少量资产「实例化/注册到场景」（如每帧 2-4 个对象，控制在 2ms 内），大块资产拆成多次提交 ③**帧预算熔断**——单帧流式加载耗时超过预算（如 3ms）就暂停，剩余推迟到下一帧。这样无论块多大，玩家都感受不到 hitch。

### 📖 深度展开

**1. 网格分块与加载圈示意**

```
              玩家移动方向 →
   ... ┌──┬──┬──┬──┬──┬──┐ ...
   ... │卸│卸│载│玩家│载│载│ ...   内圈(常驻)：玩家所在块 + 8 邻块
   ... ├──┼──┼──╳──┼──┼──┤ ...   外圈(缓冲)：提前异步加载
   ... │卸│待│载│ 载 │载│载│ ...   圈外：待卸载(LRU)
   ... └──┴──┴──┴──┴──┴──┘ ...
        ▲              ▲
        卸载队列        预加载队列（按到玩家距离排序，近的优先）

加载优先级：玩家正前方块 > 侧前方 > 侧方 > 后方
卸载策略：  离开加载圈 → 进 LRU 队列 → 内存紧张时真正释放
```

**2. 流式加载的帧内流水线**

```
┌─ 后台 IO 线程 ──────────────────────────────────┐
│  读取待加载块的资产文件 → 解压 → 解析到内存结构   │  与主线程并行，无阻塞
└───────────────┬────────────────────────────────┘
                │ 加载完成的资产进「就绪队列」
┌───────────────▼────────────────────────────────┐
│  主线程（每帧固定预算 ~3ms）                      │
│  从就绪队列取 N 个 → 实例化/注册到场景/烘焙碰撞    │  分帧提交，控制单帧耗时
│  超预算则暂停，剩余推迟                          │
└────────────────────────────────────────────────┘
   结果：玩家始终在已加载完的世界里移动，加载在背后悄悄进行
```

**3. Unity 的 World Partition（UE 同名机制）实现思路**

```csharp
// 简化的流式加载调度器
public class WorldStreaming : MonoBehaviour {
    public float cellSize = 256f;        // 块边长
    public int loadRadius = 3;           // 加载圈半径（块数）
    private readonly Dictionary<Vector2Int, StreamedCell> _active = new();
    private readonly PriorityQueue<StreamedCell> _ready = new(); // IO 完成的待提交

    void Update() {
        Vector2Int playerCell = WorldToCell(transform.position);
        // 1. 计算需要的块 vs 当前激活的块 → 差集
        var wanted = CellsInRadius(playerCell, loadRadius);
        foreach (var c in wanted.Except(_active.Keys))
            if (!_loading.Contains(c)) StartAsyncLoad(c);   // 后台加载
        foreach (var c in _active.Keys.Except(wanted))
            ScheduleUnload(c);                              // 进卸载队列

        // 2. 分帧提交：每帧只实例化预算内的块
        float budget = 3f / 1000f; // 3ms
        while (_ready.Count > 0 && Time.realtimeSinceStartup - _frameStart < budget) {
            var cell = _ready.Dequeue();
            cell.Instantiate();     // 注册到场景（主线程操作）
            _active[cell.coord] = cell;
        }
    }
    // 关键：StartAsyncLoad 在 IO 线程读盘+解压，完成后入 _ready 队列，
    //       主线程只做轻量的「实例化」，避免同步 IO 卡帧。
}
```

**4. 内存预算与卸载策略**

```
内存预算 = 总可用 - 安全余量（如 4GB - 512MB = 3.5GB 给流式块）
  每块平均占用 ~20MB（地形+物体+碰撞+NavMesh）→ 最多常驻 ~175 块

当活跃块内存 > 预算：
  ① 按 LRU 挑出最久未访问的圈外块
  ② 检查引用计数（场景里还有实体引用？等引用归零）
  ③ 卸载：销毁实例 → 释放资产引用 → 归还内存
  ⚠️ 陷阱：NavMesh、共享材质、ScriptableObject 可能被多块引用，
      必须引用计数归零才能真释放，否则「看似卸载实则泄漏」→ OOM
```

**5. 流式加载 vs 离散场景切换对比**

| 维度 | 流式加载（Streaming） | 离散场景切换 |
|------|-------------------|------------|
| 世界形态 | 连续大世界，无边界感 | 独立场景，有 loading 界面 |
| 加载时机 | 玩家移动时增量加载 | 切场景时原子加载 |
| 玩家体验 | 无缝、无读条 | 有 loading screen |
| 内存管理 | 动态进出、LRU 卸载 | 整场景换入换出 |
| 适用类型 | 开放世界（GTA/原神大世界） | 关卡制（马里奥/秘境副本） |
| 实现复杂度 | 高（调度+预算+引用计数） | 中 |
| 典型代表 | World Partition、SubScene | LoadScene、Addressables |

二者常**混合**：大世界用流式加载无缝漫游，进入「秘境/副本」时用离散切换（此时 loading 是可接受的）。

### ⚡ 实战经验

- **预加载距离必须 > 最大移速 × 单块加载耗时，否则「走到虚空」**：玩家骑坐骑时速 30m/s，单块（256m）后台加载要 0.5s，则玩家 0.5s 能移动 15m——预加载圈至少要覆盖「当前块 + 缓冲带 > 15m」。算错这个，玩家就会看到「前方地形突然弹出」或「踩到还没加载的区域掉出世界」。坐骑/传送等高速移动场景要动态扩大预加载半径。
- **同步实例化是 hitch 的头号元凶，必须分帧**：哪怕资产已在内存，把几千个 GameObject 实例化、注册碰撞、构建 NavMesh 一次做完，主线程照样卡几百 ms。务必拆成「每帧只实例化 N 个」的流式提交，并用 `Time.realtimeSinceStartup` 严格卡预算。这部分常被忽略——「异步加载完了为什么还卡？」答案就是同步实例化没分帧。
- **引用计数是防内存泄漏的生命线**：流式块共享大量资产（通用材质、NavMesh 片段、ScriptableObject 配置），简单「卸载块就释放」会误删还在被其他块用的资产，或反过来该释放的因残留引用而泄漏。每个资产维护引用计数，块加载时 +1、卸载时 -1，归零才真释放。上线前必须接内存监控（performance-monitoring），长时间游玩后内存曲线应该是「锯齿平稳」而非「单调上涨」。
- **跨块实体（如横跨两块的长桥、河流）要专门处理**：实体只属于一个块的话，跨块物体会随某块卸载而消失/闪烁。方案：①大物体归属「关键块」（玩家中心块）常驻 ②用「逻辑实体 + 多块引用」解耦 ③UE 的 World Partition 用 Data Layer 把跨块物体重定向。这块设计漏了，开放世界就会出现「桥走到一半消失」的穿帮。
