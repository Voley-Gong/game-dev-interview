## 参考答案

### ✅ 核心要点

1. **问题本质：实体规模 × tick 频率 = CPU 预算爆炸**。一个开放世界可能同时存在上万 NPC/怪物/可交互物，若每个都每帧（60Hz）完整 tick（AI 决策 + 动画 + 物理），CPU 瞬间打满。但玩家同一时刻真正「关注」的只是屏幕内及周边的少数实体——远处的山鸡、看不见的商店 NPC，每帧更新毫无感知价值。模拟 LOD 的核心就是**按「玩家感知价值」给实体分配不同的更新频率**。

2. **模拟 LOD（Simulation LOD）= 给实体分「频率档位」**：典型分三档——**Tier0（满帧 60Hz）**：屏幕内、玩家附近、战斗中的实体；**Tier1（降频 10-20Hz）**：中距离、可见但不焦点；**Tier2（冻结/事件驱动）**：远处、不可见，只在被攻击/玩家接近时才唤醒。把万级实体的「每帧完整 tick」变成「少数满帧 + 多数降频 + 大多数冻结」，总 CPU 开销骤降一个数量级。

3. **Time-slicing（时间片轮转）是降频的标准实现**：不降低单次更新的工作量，而是「每 N 帧才更新一次」。把实体按 Tier 分桶，Tier1 的实体轮流在「分配到的那一帧」更新——100 个 Tier1 实体分到 6 帧轮转，平均每个 10Hz 但每帧只算 ~17 个，把峰值摊平。关键是**轮转要均匀打散**（抖动），避免某帧集中更新 100 个造成帧时间尖刺。

4. **降频带来的「跨帧状态陈旧」必须补偿**：一个 10Hz 更新的怪物，玩家在它两次更新之间高速冲过去，会出现「瞬移感」或「判定延迟」。解决：①**插值/外推**——渲染层用上一帧速度外推位置，视觉平滑 ②**关键事件即时唤醒**——被攻击、进入视野时临时升档到满帧，保证响应性 ③**确定性步长**——降频逻辑用固定 dt 推进，避免帧率波动导致行为漂移。

5. **「事件驱动 + 频率分层」是冻结层（Tier2）的底层模型**：远处/不可见实体不主动 tick（彻底冻结、零开销），而是注册到「触发器」（玩家进入范围、时间到了、被广播伤害）。平时 0 CPU，事件来了才计算。这是把「轮询」换成「通知」的典型优化，和场景四叉树/触发器体积配合，实现「只有相关的东西才算」。

### 📖 深度展开

**1. 频率分层的「金字塔」与 CPU 预算分配**

```
                实体数        tick 频率     单帧实际计算
   Tier0 满帧    ~50 个        60 Hz        50 × 1 = 50 次/帧   ← 屏幕内/战斗中
   Tier1 降频    ~500 个       10 Hz        500/6 ≈ 83 次/帧    ← 中距离可见
   Tier2 冻结    ~10000 个     0 Hz(事件)   ~0 次/帧            ← 远处/不可见
                                                      ↓
   总计 ≈ 133 次/帧  vs  全满帧 10550 次/帧  →  降了 ~80 倍
```

**2. Time-slicing 分桶轮转实现**

```csharp
public class SimulationLOD {
    // 每个 tier 一个桶，桶内实体按 round-robin 分到「更新槽」
    private readonly List<Entity>[] _buckets = {
        new(), // Tier0
        new(), // Tier1
        new()  // Tier2（实际用事件驱动，这里只是占位）
    };
    // Tier1 每 6 帧轮一圈 → 平均 10Hz
    private const int TIER1_STRIDE = 6;

    public void Tick(int frameIndex) {
        // Tier0：每帧全更新
        foreach (var e in _buckets[0]) e.Update(dt: 1f/60f);

        // Tier1：按 frameIndex % STRIDE 取本帧该更新的子集（均匀打散）
        int slot = frameIndex % TIER1_STRIDE;
        // 用更大的 dt 补偿降频（10Hz → dt=1/10），保证逻辑步长一致
        for (int i = slot; i < _buckets[1].Count; i += TIER1_STRIDE) {
            _buckets[1][i].Update(dt: 1f / 10f);
        }
        // ⚠️ 关键：均匀打散，避免某帧 slot 命中太多实体造成帧时间尖刺
    }
}
```

**3. 视觉插值：降频更新 + 满帧渲染的配合**

```
逻辑层（降频 10Hz）：        渲染层（满帧 60Hz）：
  t=0.0  pos=0      ─┐
  t=0.1  pos=1       ├─ 逻辑每 100ms 跳一次
  t=0.2  pos=2      ─┘
                       渲染在两次逻辑更新之间，用上一帧速度外推：
                       t=0.00 render=0.0
                       t=0.02 render=0.2  ← 外推：lastPos + vel*0.02
                       t=0.04 render=0.4
                       ...
                       t=0.10 render=1.0  ← 逻辑刚更新，对齐，无瞬移
  结果：玩家看到的是平滑的 60fps 移动，背后逻辑只算 10 次。
```

**4. 分层决策：什么决定实体在哪个 Tier**

```
              ┌── 屏幕内 / 玩家半径 < 10m / 战斗中 ──▶ Tier0（满帧）
实体评估 ─────┤
              ├── 可见但远 / 半径 10-50m          ──▶ Tier1（降频）
              │
              └── 不可见 / 极远 / 无交互可能        ──▶ Tier2（冻结+事件）

评估频率本身也要控制：不能每帧给万个实体重新算 tier。
做法：用空间分区（四叉树/Grid）+ 玩家移动触发重评估，
     或每 N 帧（如 30 帧）批量重分级一次，期间保持稳定。
```

**5. 频率分层 vs 多线程并行：互补而非替代**

| 维度 | 模拟 LOD（频率分层） | 多线程 Job System |
|------|-------------------|------------------|
| 优化思路 | 减少「总工作量」 | 把固定工作量分摊到多核 |
| 适用场景 | 实体多但关注点集中 | 单帧工作量大、可并行 |
| 效果上限 | 降 1-2 个数量级 | 受限于核数（4-8x） |
| 副作用 | 降频带来插值/延迟 | 线程同步/数据竞争 |
| 关系 | **两者叠加**：先 LOD 减总量，再 Job 并行剩余 |

成熟引擎都是「先按 LOD 砍掉 90% 无效更新，再用 Job System 并行跑剩下的 10%」。只并行不减量，万级实体照样卡。

### ⚡ 实战经验

- **降频的 dt 必须用「累计的真实时间」而非「固定 1/频率」**：用固定 dt 会导致实体行为与现实时间脱钩（卡顿时实体按固定步长跑，和满帧实体不同步）。正确：降频实体每次更新时，dt = 距离上次更新的真实流逝时间，保证「每秒位移量」和满帧实体一致，避免「降频怪物变慢/变快」的诡异 Bug。
- **关键交互事件必须「即时升档」**：一个 Tier2 冻结的远处怪物被玩家远程技能打中时，若还按事件队列延迟唤醒，会出现「打中了没反应、几秒后才掉血」。任何「被攻击/进入战斗/被对话」的事件要立即把实体临时升到 Tier0 满帧，事件结束后再降回去。漏掉这条规则会导致战斗手感崩坏。
- **分桶轮转要防「共振尖刺」**：如果所有 Tier1 实体都从 slot=0 开始，第 0、6、12...帧会集中更新，其他帧空闲，帧时间呈锯齿状尖刺。初始化时给每个实体随机分配起始 slot（jitter），把更新负载均匀打散到每一帧。配合帧时间监控（performance-monitoring）验证负载是否平滑。
- **别对「需要确定性的逻辑」降频**：帧同步/回放游戏里，所有玩家必须跑相同频率才能结果一致，对部分实体降频会导致 desync。模拟 LOD 只适用于**渲染表现层和「与服务端权威无关的客户端模拟」**（如远景 NPC 的闲逛动画），绝不能用于影响公平判定（命中/伤害）的逻辑——那些必须满帧且由服务端或确定性循环驱动。
