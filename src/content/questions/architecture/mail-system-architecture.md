## 参考答案

### ✅ 核心要点

1. **邮件类型与来源分离**：系统邮件（全服公告 / 维护补偿，服务端批量生成）、个人邮件（玩家互发，C2C）、公会邮件、战斗结算邮件来源不同。不同来源走不同生成管线，但统一存储与展示模型，避免每类一套代码带来的维护灾难。

2. **附件事务性发放（核心难点）**：邮件附件（道具 / 货币）必须保证「要么全发、要么不发」，防止单个附件发放成功但玩家中途离线导致部分丢失。采用预写入附件列表 → 逐个发放 → 全部成功才标记已领取 → 失败回滚的状态机，任何环节崩溃可重试且幂等。

3. **过期与容量清理**：邮件有有效期（如 30 天 / 未读 7 天系统清理），且邮箱有容量上限（如 100 封）。过期清理用后台定时扫描 + 分批删除；容量超限时新邮件走「拒收或挤掉最旧」，避免无限增长拖垮存储与查询性能。

4. **未读状态与推送**：客户端维护未读计数，新邮件到达通过长连接推送。推送与拉取分离——推送只通知「有新邮件」，具体内容由客户端按需拉取，避免在推送通道传输大附件数据造成带宽浪费和卡顿。

5. **分页加载与冷热分离**：玩家可能有上千封历史邮件，首次只加载首页（N 封），滚动到底触发分页拉取。已领取 / 已读的旧邮件可下沉到冷存储或归档表，热表只保留近期未处理邮件，保证首页查询的响应速度。

6. **跨服与跨账号投递**：MMO 跨服邮件需通过中心邮件服务路由，按目标服务器 ID 投递。账号级邮件（如通行证奖励）要支持多角色共享，需明确领取所有权（角色级 vs 账号级）防止同一奖励被重复领取。

### 📖 深度展开

#### 附件事务性发放状态机

附件发放是邮件系统最容易出 Bug 的地方。玩家点击「领取」后，网络抖动、服务崩溃、背包满都可能导致「部分发放」的脏数据。这类问题在生产环境极难复现，往往要等到玩家投诉「领了没收到」或「道具发重」才暴露，事后排查成本极高。因此必须用状态机约束整个生命周期，并保证每一步可重试、可回滚、可幂等。

```
邮件生命周期：
  CREATED(已生成,含附件列表)
     ↓ 服务端投递
  DELIVERED(已送达,可领取)
     ↓ 玩家点击领取
  [发放附件: 逐个发,全部成功]
     ├─ 全部成功 → CLAIMED(已领取,附件清空)
     └─ 任一失败 → 回滚,保持 DELIVERED(可重试)
  DELIVERED/CLAIMED 超过有效期 → EXPIRED(过期,未领附件销毁)
```

下面是幂等领取的实现，关键点：先查状态守卫、预校验背包、逐个发放、失败回滚。

```typescript
// 附件事务性领取：幂等 + 预校验 + 回滚
async function claimAttachments(mailId: number, playerId: number) {
  const mail = await getMail(mailId, playerId);
  // 幂等：已领取直接返回成功（防网络重发）
  if (mail.status === 'CLAIMED') return { ok: true, msg: '已领取' };
  if (mail.status !== 'DELIVERED') throw new Error('不可领取');

  // 预校验背包空间，防止发到一半背包满
  if (!hasInventorySpace(playerId, mail.attachments)) {
    throw new Error('背包空间不足，请清理后重试');
  }

  // 逐个发放，全部成功才标记；任一失败回滚
  const granted: Attachment[] = [];
  try {
    for (const att of mail.attachments) {
      await grantItem(playerId, att);   // 单个发放（自身也需幂等）
      granted.push(att);
    }
    await setMailStatus(mailId, 'CLAIMED'); // 全部成功，原子标记
    return { ok: true };
  } catch (e) {
    await rollbackGranted(playerId, granted); // 回滚已发的
    throw e;  // 保持 DELIVERED，玩家可重试
  }
}
```

| 幂等设计要点 | 具体做法 | 作用 |
| --- | --- | --- |
| 幂等键 | `mailId + slot` 联合唯一 | 同一附件不会被发放两次，重发请求命中已发记录 |
| 状态机守卫 | 进入 `claimAttachments` 先判 `CLAIMED` 直接返回 | 防止并发请求或网络重发触发重复发放 |
| 回滚策略 | 逐个撤销已发放的 `granted` 列表 | 部分失败时恢复到发放前状态，保证「全发或全不发」 |

#### 邮件存储模型与分页加载

邮件数据用「主表 + 附件子表」分离，主表一封一行便于分页查询，附件子表一对多承载多道具。状态字段区分领取进度，过期时间字段驱动后台清理。

这种拆分的好处是：主表行窄、字段稳定，适合建复合索引（如 `playerId + id DESC`）支撑游标分页；附件变化时只更新子表，不触碰主表，减少锁竞争。同时附件表与主表解耦后，可以独立做批量清理——过期邮件直接删主表行并级联清理附件，不会因附件过多拖慢单行删除。

```typescript
// 邮件主表：一封邮件一行
interface MailRecord {
  id: number;                  // 邮件ID（主键）
  playerId: number;            // 收件人
  type: 'system' | 'personal' | 'guild' | 'combat';
  title: string;
  body: string;
  senderName: string;
  createdAt: number;           // 创建时间戳
  expireAt: number;            // 过期时间戳
  status: 'DELIVERED' | 'CLAIMED' | 'EXPIRED';
  isRead: boolean;
}
// 附件子表：一封邮件多个附件（一对多）
interface MailAttachment {
  mailId: number;
  slot: number;                // 槽位序号（幂等键之一）
  itemType: 'item' | 'currency';
  itemId: number;
  count: number;
}
```

历史邮件上千封时，`OFFSET` 深翻页会随页码增大扫描越来越多的行，性能急剧退化。改用基于 `id` 的游标分页，每页查询复杂度恒定：

游标分页的另一个优势是稳定性：无论玩家翻到第几页，每次查询都走索引定位（`playerId + id < cursor`），扫描行数固定为 `LIMIT` 大小，不会因数据总量增长而劣化，适合邮件这类只会累积增长的数据。

```
分页游标查询（避免 OFFSET 深翻页性能退化）：
首页:  SELECT * FROM mail WHERE playerId=? ORDER BY id DESC LIMIT 20
下一页: SELECT * FROM mail WHERE playerId=? AND id < ?(上一页最后id) ORDER BY id DESC LIMIT 20
```

| 策略 | 热表查询性能 | 实现复杂度 | 适用规模 |
| --- | --- | --- | --- |
| 单表全量 | 随数据增长变慢 | 低 | 小规模 |
| 冷热分表 | 热表恒定快 | 中 | 中规模 |
| 归档表 + 分区 | 最优 | 高 | 大规模 MMO |

> 注：热表只保留 `DELIVERED` + 近期 `CLAIMED` 的邮件，旧的 `CLAIMED` / `EXPIRED` 定期迁移到冷表 / 归档表。这样首页查询永远只扫热表的少量行，响应稳定在毫秒级。
>
> 迁移时机通常选择低峰期，按 `expireAt` 或 `createdAt` 批量滚动：每轮迁移一批（如 5000 行），用后台任务分摊负载，避免一次性大事务锁表影响在线玩家的正常收发。

#### 推送与拉取解耦架构

新邮件通知要快但不能重——推送通道带宽有限，塞满附件序列化数据会导致整条推送通道卡顿。因此把「通知有新邮件」和「拉取具体内容」拆成两条链路，各司其职、互不阻塞。

这条原则的本质是「通知轻量化、数据按需化」：推送只负责「捅一下」客户端，真正的数据走显式 HTTP 拉取，失败可重试、可缓存。这样即使玩家同时收到几十封系统邮件，推送通道也不会被撑爆。

```
服务端                                  客户端
┌─────────────┐    长连接推送        ┌──────────────┐
│ 邮件生成     │ ──{type:newMail}──→ │ 收到推送      │
│ 服务        │   (仅含 count)       │ → 刷新未读计数 │
└─────────────┘                      │ → 按需拉取首页  │
        ↑                            └──────────────┘
        │ 断线重连补偿拉取
        │  GET /mail/unread?since=本地计数
        └────────────────────────────┘
```

推送只携带计数，不携带附件数据，原因有三：一是推送通道带宽紧张，大附件包容易阻塞；二是完整邮件序列化成本高、增加 CPU 开销；三是玩家可能不在邮件界面，全量推送反而造成卡顿。断线重连补偿机制是关键兜底：客户端每次重连后用本地未读计数与服务端差值对比，主动拉取漏掉的新邮件，避免因弱网丢包导致玩家「永远不知道有新邮件」。

更进一步，补偿拉取要带上 `since` 游标（本地最后已知计数或时间戳），只拉取增量而非全量未读列表，保证补偿请求本身也轻量。即使连续断线重连多次，每轮也只补差值，不会重复拉取已读邮件。

| 推送内容 | 带宽开销 | 可靠性 |
| --- | --- | --- |
| 全量邮件（含附件） | 高 | 低——易丢大包 |
| 仅计数 | 极低 | 中——需补偿拉取 |
| 计数 + 摘要 | 低 | 中 |

### ⚡ 实战经验

1. **附件发放幂等**：玩家点击领取后网络抖动重发请求，若无幂等校验会发两份道具。用 mailId + 附件 slot 作为幂等键，已 CLAIMED 状态直接返回成功，实测杜绝了约 90% 的「道具发重」客诉。

2. **全服邮件性能**：全服 10 万玩家发补偿邮件，若逐条插入 DB 会卡几分钟阻塞业务。用批量 INSERT + 分批 commit（每 1000 条一批）并在低峰期预生成，耗时从约 15 分钟降到 20 秒。

3. **附件领取与背包满**：玩家背包满时领取附件不能直接发放（会溢出丢失）。应预校验背包空间，空间不足时保留附件、不标记已领取，提示玩家清理背包后重试，避免「领了没收到」的硬伤。

4. **过期清理竞态**：后台清理过期邮件时玩家正在领取同一封，会出现清理了正在发放的附件。给「未领取且有附件」的邮件加额外宽限期（过期后再保留 7 天），或领取时加分布式锁。

5. **推送丢失补偿**：长连接推送可能因断线 / 弱网丢失，客户端每次重连应主动拉取未读计数差值（本地 vs 服务端）补偿漏掉的「新邮件」通知，避免玩家永远不知道有新邮件。
