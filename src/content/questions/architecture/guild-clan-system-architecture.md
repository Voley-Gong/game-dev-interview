## 参考答案

### ✅ 核心要点

1. **数据模型三层分离**：GuildDef（配置：公会等级上限、职位定义、科技树）与 GuildInstance（运行时：公会实例、成员、仓库、日志）分离，运行时数据再按热度分表——主表存基础信息（公会 ID/名字/等级）、成员表存成员关系、仓库表存资产、日志表冷归档。单个公会成员上限通常 100~500 人，超大规模（MMO 万级公会）需按公会 ID 分片。

2. **RBAC 权限分级模型**：采用角色（Role）+ 权限点（Permission）的 RBAC 模型而非硬编码职位判断。标准职位链：会长（Owner）→副会长（ViceLeader）→精英（Elite）→普通成员（Member）→见习（Trainee），每级用权限位图或权限点集合定义能做什么（踢人/审批/发公告/调仓库/开公会战）。支持自定义职位满足策划差异化需求。

3. **成员状态机驱动生命周期**：成员从「申请入会」→「审核通过」→「正式成员」→「晋升/降级」→「退会/踢出」走完整状态机。退会/踢出采用软删除（保留 7~14 天贡献记录），便于撤销误操作和申诉核查；转服、公会合并等跨数据操作必须走事务，保证贡献度、职位、仓库资产原子性迁移。

4. **公会仓库与捐献事务**：捐献是高频写操作，必须事务化——扣减个人资源 → 增加公会仓库 → 记录贡献度 → 三步原子提交。高并发场景（晚高峰单公会 200+ 次/秒捐献）下，数据库乐观锁冲突率超 30% 时需降级为 Redis Lua 原子脚本，将 TPS 从几百提升到数千。

5. **公会科技/被动增益树**：公会科技是全体成员共享的被动增益（如「全员攻击力 +5%」），通过成员捐献资源升级科技节点。科技等级解锁后需向在线成员广播增益下发，离线成员登录时按当前科技快照补算。注意双来源防重复加成（个人装备 + 公会科技都加攻击时，要用 Modifier 链区分来源）。

6. **跨服公会战赛季架构**：公会战是高并发团队玩法，匹配池 + 赛季积分制 + 战场实例隔离。战场内 50v50~100v100 团战的状态同步必须走兴趣区域管理（Area of Interest），只同步视野内实体，否则带宽爆炸。分布式锁防止同一公会重复参战或刷分。

### 📖 深度展开

#### 1. 数据模型与分表策略

公会系统的核心矛盾：单个公会的数据强相关（成员、仓库、日志都要按公会 ID 聚合查询），但全服公会数量大（万级），不能单表存储。

```typescript
// 公会配置（静态，策划维护）
interface GuildDef {
  guildId: number;
  name: string;
  maxMembers: number;       // 成员上限，如 100/300/500
  roleDefs: RoleDef[];      // 职位定义，含权限位图
  techTreeId: number;       // 关联的科技树配置
}

// 公会运行时实例（热数据，主表）
interface GuildInstance {
  guildId: number;
  name: string;
  level: number;
  leaderId: number;         // 会长角色 ID
  fundContribution: number; // 公会资金（捐献累计）
  createdAt: number;
  version: number;          // 乐观锁版本号
}

// 成员关系（成员表，按 guildId 分表）
interface GuildMember {
  guildId: number;          // 分片键
  roleId: number;           // 玩家角色 ID
  role: GuildRole;          // 当前职位枚举
  contribution: number;     // 个人累计贡献度
  joinedAt: number;
  status: MemberStatus;     // Active | SoftDeleted | Suspended
  permissions: bigint;      // 权限位图，位运算判定
}
```

数据分表策略对比：

| 数据类型 | 存储方案 | 访问模式 | 理由 |
|---------|---------|---------|------|
| 公会主信息 | 主表（全服单表或按 ID 分片） | 高频读、低频写 | 查询公会名字/等级极频繁 |
| 成员列表 | 按 guildId 分表（如 1024 分片） | 按公会聚合查询 | 避免大表 OFFSET 翻页 |
| 公会仓库 | 独立表 + guildId 索引 | 中频读写 | 物品堆叠需独立事务 |
| 操作日志 | 冷归档表（按月分区） | 低频读、仅审计 | 数据量大但极少查询 |
| 公会战战绩 | 独立赛季表 | 赛季结束归档 | 赛季数据隔离 |

成员超 1000 人时，分页查询必须用游标分页（`WHERE guildId=? AND roleId > ? LIMIT 50`）而非 OFFSET——OFFSET 10000 需扫描 10000 行，查询从 2ms 飙升到 800ms。

#### 2. RBAC 权限模型与成员状态机

```typescript
// 权限位图定义（每位代表一个权限）
enum Permission {
  KICK          = 1n << 0n,
  APPROVE_JOIN  = 1n << 1n,
  POST_NOTICE   = 1n << 2n,
  MANAGE_BANK   = 1n << 3n,
  START_GUILD_WAR = 1n << 4n,
  PROMOTE       = 1n << 5n,
}

// 角色定义：职位 → 权限位图
const ROLE_PERMISSIONS: Record<GuildRole, bigint> = {
  Owner:      0b111111n,   // 全权限
  ViceLeader: 0b101111n,   // 除晋升外全权限
  Elite:      0b001100n,   // 公告 + 仓库
  Member:     0b000000n,   // 仅基础权限
  Trainee:    0b000000n,   // 见习无管理权
};

function hasPermission(member: GuildMember, perm: Permission): boolean {
  return (member.permissions & perm) === perm;
}
```

成员生命周期状态机：

```
        ┌──────────┐  审核通过   ┌──────────┐
   申请 │ APPLIED  │ ─────────→ │  MEMBER  │ ← 入会
        └──────────┘            └────┬─────┘
            │ 拒绝                    │ 晋升/降级
            ↓                    ┌────┴─────┐
        ┌──────────┐  退会/踢出  │ 状态变更  │
        │ REJECTED │ ←────────── │(职位变更) │
        └──────────┘             └────┬─────┘
                                      │ 退会/踢出
                                      ↓
                                ┌──────────────┐
                                │ SOFT_DELETED │ ← 7~14天宽限期
                                │ (可撤销/申诉) │
                                └──────┬───────┘
                                  超期 │ 清零贡献
                                      ↓
                                ┌──────────┐
                                │ DELETED  │ ← 彻底移除
                                └──────────┘
```

会长转让是高危事务：必须校验目标成员在职时长 ≥ 阈值（防恶意夺权）、目标在线确认、原会长降为副会长，全程写审计日志。

#### 3. 公会战赛季架构与并发控制

```
赛季时间轴：
┌─────────┬──────────────┬──────────────┬───────────┐
│ 报名期   │   小组赛      │   淘汰赛     │  结算期    │
│(1天)    │ (7天, 循环赛) │ (3天, 单败)  │ (1天)     │
└─────────┴──────────────┴──────────────┴───────────┘
     ↓          ↓               ↓             ↓
  锁定名单   积分累积         晋级/淘汰     发奖+归档
```

公会战匹配与战场实例隔离：

```
   公会A(50人) ──┐
                 ├──→ 匹配池 ──→ 分配战场实例 Instance#1024
   公会B(50人) ──┘                    ↓
                                   独立服务器进程
                                   (资源/状态隔离)
                                        ↓
                                   结算 → 积分写入赛季表
                                   → 奖励发放(事务)
```

防止重复参战的分布式锁方案对比：

| 方案 | 实现 | 适用场景 | 代价 |
|------|------|---------|------|
| Redis SETNX | `SET guild:war:lock:{guildId} NX EX 7200` | 单次战斗锁定 | Redis 故障则失效 |
| Redlock | 多 Redis 节点多数派 | 高可用要求 | 复杂度高，争议方案 |
| ZooKeeper | 临时顺序节点 + Watch | 强一致性 | 运维重，性能一般 |
| 数据库行锁 | `SELECT ... FOR UPDATE` | 简单场景 | 高并发下死锁风险 |

实战中常用 Redis SETNX + 数据库唯一约束（guildId + 赛季号）双保险——Redis 做快速互斥，数据库做最终一致性兜底。

### ⚡ 实战经验

1. **百万级成员分页查询**：主表只存公会基础信息（公会 ID/名字/等级，单行约 200B），成员列表按 guildId 分表并用游标分页。早期版本用 `LIMIT 50 OFFSET 10000`，查询从 2ms 涨到 800ms；改用 `WHERE roleId > lastId LIMIT 50` 索引范围扫描后稳定在 3ms。

2. **捐献并发高峰降级**：晚高峰 20:00~22:00 单公会每秒 200+ 次捐献，数据库乐观锁 version 字段冲突率超 30%（大量重试拖垮 DB）。降级为 Redis Lua 原子脚本（扣个人 + 加公会仓库 + 记贡献在一个脚本内原子完成），TPS 从 200 提升到 5000+，再异步批量回写数据库。

3. **公会战带宽优化**：战场内 100v100 团战，若每帧全量同步所有实体状态（100 人 × 4KB/帧 = 400KB/帧，60fps = 24MB/s），客户端带宽直接爆掉。改用兴趣区域管理（AoI），只同步玩家视野半径内的约 20 个实体，带宽降到约 2.4MB/s，降幅 90%。

4. **退会贡献清零的客诉治理**：玩家退会后公会科技贡献是否清零是策划核心决策点。采用 7 天宽限期（可撤销退会），超期清零时强制写审计日志（roleId/guildId/清零贡献值/时间戳）。上线审计日志后，日均处理相关客诉从 8~12 单降到 1~2 单（有据可查，不再扯皮）。

5. **公会合并事务边界**：两个公会合并（A 并入 B）是极复杂的跨数据事务——A 的成员全部转为 B 成员，A 的仓库资产转移到 B，A 的贡献度保留但职位重置。必须全量快照 + 灰度执行（先转移主信息，再分批转移成员），单次合并事务耗时控制在 30 秒内，超时回滚。曾因未做超时回滚导致一次合并卡死，影响 300+ 玩家长达 2 小时。
