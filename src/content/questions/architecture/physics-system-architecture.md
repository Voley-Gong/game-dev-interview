## 参考答案

物理系统是游戏引擎中耦合最重、性能瓶颈最敏感的子系统之一。面试官问到"物理系统架构怎么设计"，往往不是想听你怎么调 `AddForce`，而是考察你对步进时序、碰撞管线、刚体分层、查询抽象和确定性约束的整体把握。下面从核心要点、深度展开、实战经验三个层面拆解。

一个成熟的物理系统架构通常由以下几个职责清晰的模块构成：

- **步进调度层**：管理固定步长时间累积、子步拆分、渲染插值，决定物理以什么节奏推进。
- **碰撞管线层**：宽相位 → 窄相位 → 求解器三阶段流水线，是性能与精度的核心。
- **刚体抽象层**：Dynamic / Kinematic / Static / Trigger 四种角色的统一建模。
- **查询服务层**：对外暴露 Raycast / Overlap 等只读查询，供战斗、AI、UI 复用。
- **后端适配层**：把上述抽象映射到具体引擎（PhysX / Havok / Box2D / 自研），支持切换。

### ✅ 核心要点

1. **固定步长时间步进 (Fixed Timestep)**：物理必须以恒定步长（如 1/60s）推进，与渲染帧率解耦。采用累加器（Accumulator）模式，渲染帧将剩余 dt 累积，每达到一个 fixedStep 就执行一次物理步进，既避免 spiral of death（死亡螺旋，步进赶不上真实时间），又保证数值积分稳定性和网络帧同步的确定性。

2. **三阶段碰撞检测管线 (Broadphase → Narrowphase → Solver)**：宽相位用 AABB Tree / Sweep-and-Prune / 均匀网格快速剔除不可能碰撞的对（O(n log n) 或 O(n)）；窄相位用 GJK / SAT 精确计算接触点和法线；求解器用 Sequential Impulse / PGS 迭代求解约束和穿透分离。三阶段职责分离是性能关键——宽相位决定候选对数量，窄相位和求解器决定精度。

3. **刚体/运动学体/触发器的角色分层**：Dynamic 刚体受完整力学影响（重力/碰撞响应）、Kinematic 手动控制位置但不被力推动、Static 无限质量仅作碰撞源、Trigger 只检测重叠不产生物理响应。正确选型能避免角色控制器被爆炸力推飞或弹球穿透薄墙这类经典 Bug。

4. **连续碰撞检测 (CCD) 防穿透**：离散采样下高速物体（子弹）会跨越薄墙产生隧道穿透（Tunneling）。CCD 用扫掠形状（Swept AABB / Sphere Cast）在前后两帧之间插值检测碰撞，代价是更高的 CPU 开销，需按对象类型（高速抛射物层）选择性开启。

5. **查询抽象层 (Raycast/SphereCast/Overlap)**：物理系统对外提供统一的射线检测、球形扫掠、重叠查询接口，屏蔽底层引擎差异。战斗命中判定、UI 拾取、AI 视野检测都通过这层而非直接访问碰撞体，便于切换物理后端（PhysX / Havok / Box2D / 自研引擎）。

6. **网络确定性与浮点量化**：帧同步要求物理演算逐帧确定，需锁定物理后端版本、禁用非确定性特性（多线程求交顺序、硬件相关浮点 FMA）、用定点数或量化浮点。状态同步则可放松本地物理演算，但要服务端权威校正客户端漂移。

### 📖 深度展开

核心要点回答的是"是什么"，深度展开回答的是"为什么这么设计"以及"工程上具体怎么做"。下面三个子章节分别对应物理系统最常被追问的三块：步进时序、碰撞管线、刚体与穿透——每一块都是面试官深挖细节的高频区。

#### 固定步长累加器与帧率解耦

物理步进必须以固定步长推进，而渲染帧率是可变的（掉帧、高刷屏 VRR 都会改变 dt）。如果直接把渲染 `frameDt` 丢给物理积分器，数值稳定性会随 dt 抖动而劣化——例如 Verlet 积分在 dt=1/30 与 dt=1/120 下的误差能差一个数量级。固定步长的核心思路是用累加器（Accumulator）将渲染帧的剩余时间累积，每攒满一个固定步长就执行一次物理更新，并在渲染时用插值因子平滑过渡，避免视觉抖动。

```typescript
// 固定步长累加器：物理以恒定步长推进，与渲染帧率解耦
class FixedTimestepPhysicsLoop {
  private readonly fixedDt = 1 / 60;     // 物理步长 16.67ms
  private accumulator = 0;                // 累加器
  private maxSubSteps = 5;                // 防止死亡螺旋

  // 每一渲染帧调用
  update(frameDt: number) {
    this.accumulator += frameDt;
    let steps = 0;
    while (this.accumulator >= this.fixedDt && steps < this.maxSubSteps) {
      this.stepPhysics(this.fixedDt);     // 固定步长推进
      this.accumulator -= this.fixedDt;
      steps++;
    }
    // 累加器溢出（卡顿导致）→ 丢弃剩余，避免 spiral of death
    if (steps >= this.maxSubSteps) this.accumulator = 0;

    // 插值因子：用 accumulator/fixedDt 在上一物理帧与当前物理帧间插值
    const alpha = this.accumulator / this.fixedDt;
    this.interpolateRenderTransforms(alpha);
  }
}
```

渲染帧与物理步进的时间轴对照，可以看出累加器如何在两套不同节拍之间做缓冲：

```
时间轴 →
渲染帧:  |----dt1----|---dt2---|----dt3----|  (可变)
物理步进: |--16.67--|--16.67--|--16.67--|--16.67--|  (固定)
累加器:   累积dt → 够一步就跑 → 剩余累积到下一帧
渲染插值: alpha=accumulator/fixedDt, 在两个物理状态间lerp
```

| 方案 | 稳定性 | 网络确定性 | 复杂度 |
| --- | --- | --- | --- |
| 可变步长 | 简单但不稳定（dt 抖动放大数值误差） | 不确定 | 低 |
| 固定步长 + 累加器 | 稳定 | 确定 | 中 |
| 固定步长 + 渲染插值 | 最稳（消除视觉抖动） | 确定 | 高 |

> **Spiral of death 风险**：当某帧卡顿导致 `frameDt` 远大于 `fixedDt` 时，下一帧需补跑大量子步，进一步拖慢帧率，形成越卡越补、越补越卡的死亡螺旋。`maxSubSteps` 上限通过强制丢弃溢出的累加器时间来打断这个循环，代价是一次性的"时间跳跃"——物体瞬移一小段，但帧率能稳住，这是经过业界验证的合理取舍。

值得注意的细节：渲染插值的 `alpha` 计算用的是**清空子步后的剩余累加器值**，而非原始 frameDt。这意味着插值永远落在"上一个已执行的物理帧"与"下一个待执行的物理帧"之间，渲染画面始终略滞后于逻辑状态半步以内，既平滑又不引入新的延迟感。若改用外推（extrapolation）则会因预测未来状态而在碰撞瞬间出现"穿入再弹回"的视觉瑕疵，故业界普遍采用插值而非外推。

#### 三阶段碰撞检测管线

碰撞检测不是一次性运算，而是一条分层管线：先用廉价算法剔除绝大部分不可能相交的对，再对少量候选对做精确求交，最后求解约束修正位置/速度。跳过宽相位直接做 N² 两两检测会在几百个刚体时崩盘——1000 个物体就是近百万对/帧的求交，完全不可行。

```
N 个物体
   ↓ Broadphase（宽相位）
候选碰撞对 (Candidate Pairs)   ← AABB Tree / SAP / 均匀网格，O(n)~O(n log n)
   ↓ Narrowphase（窄相位）
精确接触信息 (Contact Manifold) ← GJK / SAT，算出接触点+法线+穿透深度
   ↓ Solver（求解器）
位置/速度修正              ← Sequential Impulse / PGS 迭代求解约束
   ↓
更新刚体状态
```

宽相位方案选型直接决定整体吞吐量，下表对比四种主流方案：

| 方案 | 适合场景 | 时间复杂度 | 增删成本 |
| --- | --- | --- | --- |
| AABB Tree | 动态场景 / 中等规模 | O(n log n) | 中 |
| Sweep and Prune | 少轴运动 | O(n+k) | 低 |
| 均匀网格 | 均匀分布 2D / 大量同类体 | O(n) | 低 |
| 空间哈希网格 | 开放世界流式 | O(n) | 低 |

> **选错宽相位是物理性能的头号杀手**。例如开放世界用 AABB Tree 会导致树频繁重平衡；反之大量均匀分布的小物体用 SAP 退化为 O(n²)。宽相位本质是"用空间换时间"——网格类方案空间换时间效果好但吃内存，树类方案内存友好但动态更新贵。

窄相位两大算法各有定位：**GJK**（Gilbert-Johnson-Keerthi）用支撑函数（Support Function）迭代逼近两个凸体的最近距离，适合任意凸体、连续光滑形状，但只判断是否碰撞，需配合 EPA 算穿透深度；**SAT**（分离轴定理）将凸多边形向候选轴投影检测重叠，对 2D 多边形和简单 3D 凸包非常高效，是 2D 引擎（如 Box2D）的主力算法。求解器则将接触约束建模为 LCP（线性互补问题），用 Sequential Impulse 迭代收敛——迭代次数越多越稳但越慢，通常 4~10 次迭代是常见工程取舍。

接触流形（Contact Manifold）的管理也常被忽视：窄相位一次求交只产出一两个接触点，但刚体堆叠稳定需要多个接触点共同支撑。引擎会在多帧间持久化并积累流形点（典型 4 个），通过 clipping 维持一个稳定的接触面。堆叠场景（如箱子塔）的抖动，九成是流形管理不当或求解器迭代不足导致的。

#### 物理体类型与 CCD 选型

刚体类型选错会引发各种难以排查的 Bug，下表把四种类型的力学行为和适用场景列清楚：

| 类型 | 受力 | 碰撞响应 | 典型用途 |
| --- | --- | --- | --- |
| Dynamic 刚体 | 是 | 完整分离 + 反弹 | 弹球、可破坏物 |
| Kinematic | 否 | 仅检测不响应 | 移动平台、角色控制器 |
| Static | 否 | 仅作碰撞源 | 地形、墙壁 |
| Trigger | 否 | 仅重叠事件 | 拾取区、Buff 范围 |

高速物体的隧道穿透是离散物理的固有缺陷，CCD 通过扫掠形状沿运动轨迹插值检测来弥补：

```
离散采样（会穿透）：
  帧1: 子弹在墙左 ──→ 帧2: 子弹在墙右   （跨过了薄墙，未检测到）

CCD 扫掠检测：
  帧1 ──────Swept Sphere──────→ 帧2
         沿运动轨迹插值，在轨迹与墙相交处停下
```

> **CCD 开启代价**：CCD 需对运动轨迹做多次插值求交，单对象成本数倍于离散检测。应仅对高速对象（速度阈值如 >10 m/s）或指定层（子弹 / 抛射物）开启。全场景开 CCD 可使物理耗时接近翻倍，得不偿失。

工程上还有两种轻量替代方案值得了解：**时间戳子步（Time-of-Impact Substep）**在检测到物体本帧位移超过其最小包围盒半径时，自动对该物体单独做更细的子步拆分，成本远低于全程 CCD；**基于阈值的动态 CCD**则只在物体速度突破阈值时临时切换到扫掠检测。两者都能在精度与性能间取得比"全局开关"更细的粒度。

刚体类型与 CCD 的选型往往是面试中追问的重点：面试官可能追问"为什么角色控制器不用 Dynamic 刚体"，核心原因在于 Dynamic 会被碰撞求解器的分离冲量、斜坡摩擦、爆炸力等不可控因素推动，无法保证角色运动的确定性手感；而 Kinematic + Sweep 把移动决策权完全交回逻辑层，碰撞响应也可手动定义（贴墙滑动、台阶抬升），这正是主流引擎角色控制器的实现思路。

### ⚡ 实战经验

下面几条都是踩过坑才知道的经验，每条都附带可量化的数据，方便面试时给出有说服力的取舍依据。

1. **固定步长与渲染插值**：不插值时 60fps 物理 + 可变渲染帧会有轻微抖动。在 transform 上做插值（存储上一物理帧 transform，渲染时 lerp 到当前），实测某 3A 项目从肉眼可见抖动到丝滑，成本仅多一次 lerp。

2. **CCD 性能开销**：全场景开 CCD 可让物理耗时翻倍（实测从 2ms → 4.5ms/frame）。只对速度超阈值（如 >10 m/s）或指定层（子弹 / 抛射物）开启，其余用离散检测，性能与精度兼顾。

3. **角色控制器陷阱**：用 Dynamic 刚体做角色会被斜坡 / 台阶卡住、被爆炸力推飞。应使用 Kinematic + 手动 Sweep 移动 + 手动碰撞响应，或引擎专用 CharacterController（Unity 的 CharacterController 内部用胶囊扫掠）。

4. **触发器回调风暴**：Trigger 不产生物理响应适合拾取 / 区域检测，但海量触发器场景（如 1000 个范围 Buff 重叠）会产生大量回调。需用空间分区 + 按需启用（进入视野才激活 Trigger Collider），否则单帧回调可达上万次。

5. **Raycast 查询预算**：每帧 Raycast 过多（如 AI 视野 100 个敌人 × 多条射线）会吃掉物理预算。缓存结果 + 分帧轮询 + 降频（每 3 帧检测一次）可将物理查询耗时从 5ms 降到约 1.2ms。

> **整体取舍心法**：物理系统的设计本质是在"精度 / 性能 / 确定性"三角中取舍——帧同步电竞游戏优先确定性，3A 单机优先视觉精度，手游 MMO 优先性能预算。没有银弹，只有针对品类做对的工程权衡。

还有一点容易被忽略：**物理系统应是只写状态源，而非只读查询源**。也就是说刚体由物理引擎驱动写入 transform，但所有"我要知道某处有没有东西"的需求（命中判定、视线、寻路障碍）都走查询抽象层，绝不直接遍历碰撞体。这种读写分离能避免逻辑层与物理后端耦合，也是日后换引擎或上 ECS 时不重构的关键。

### 🔗 相关问题

把上面的知识点串起来后，可以延伸到更宏观的架构决策。下面三个问题适合作为面试的进阶追问方向，能进一步拉开候选人差距：

- 物理系统如何与 ECS 架构结合，实现大规模刚体的数据并行模拟？
- 自研物理引擎 vs 引擎内置（PhysX / Havok），什么场景值得自研、成本在哪？
- 网络游戏中客户端预测与服务端物理权威如何协作校正，避免橡皮筋抖动？

> 这三个问题分别对应**性能扩展**、**技术选型**、**网络架构**三个维度，能从单机物理架构延伸到大型在线项目的全链路设计，是资深岗位面试的高价值追问方向。
