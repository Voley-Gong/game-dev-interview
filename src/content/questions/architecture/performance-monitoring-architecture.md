## 参考答案

### ✅ 核心要点

1. **三层观测体系**：开发期剖析（引擎 Profiler / RenderDoc / Xcode）定位本地瓶颈 → 测试期自动化性能门禁（每帧预算/内存红线 CI 卡点）→ 线上遥测（APM 上报真实玩家帧率/崩溃/内存）覆盖长尾机型。
2. **核心指标四象限**：CPU（帧时间/主线程耗时/逻辑vs渲染占比）、GPU（DrawCall/三角面/显存/带宽）、内存（堆内存/资源内存/GC 频率/泄漏）、I/O（载入时间/IO 峰值）。每个指标都要有"预算（budget）"，超预算即报警。
3. **采样剖析而非全量埋点**：性能监控本身不能成为性能负担。线上用低频采样（每 N 帧采一次、每 X 秒上报一次），关键路径用环形缓冲区存最近 N 帧的耗时，崩溃/卡顿时一次性吐出，平时不传。
4. **瓶颈定位靠"分桶 + 归因"**：把每帧时间按子系统分桶（AI/物理/渲染/UI/网络），统计 P50/P95/P99 而非平均——平均值会掩盖偶发卡顿。找到最贵的那桶再下钻到具体函数。
5. **线上≠线下**：开发机是旗舰，玩家可能是低端安卓；离线 Profile 流畅不代表线上流畅。必须靠线上 APM（如腾讯灯塔/自研 SDK）拿到真实机型的帧率分布、内存水位、发热降频数据，按机型分桶分析。

### 📖 深度展开

**性能监控架构分层：**

```
┌────────────────────────────────────────────────────────┐
│  线上遥测层（APM）                                       │
│  按机型/版本聚合：帧率分布、内存水位、崩溃栈、卡顿热图    │
│  → 大盘看板 + 阈值告警（P95 帧时间 > 33ms 告警）          │
├────────────────────────────────────────────────────────┤
│  本地采集层（PerformanceCounter）                        │
│  环形缓冲区存最近 300 帧的子系统耗时                      │
│  低频采样上报，崩溃/卡顿时 dump 全量到本地                │
├────────────────────────────────────────────────────────┤
│  开发剖析层（Editor Profiler / RenderDoc / Xcode）       │
│  逐函数/逐 DrawCall 下钻，本地精确定位瓶颈               │
└────────────────────────────────────────────────────────┘
```

**核心采集模块（环形缓冲 + 分桶）：**

```csharp
// 性能计数器：按子系统记录每帧耗时，环形缓冲避免无限增长
public class PerfCounter {
    public enum Tag { Render, Logic, AI, Physics, UI, Network, IO }

    const int FRAME_BUF = 300; // 存最近 300 帧（5秒@60fps）
    float[,] frameCost = new float[FRAME_BUF, EnumCount<Tag>()];
    int head = 0;

    public IDisposable Scope(Tag tag) {  // using(Perf.Scope(Tag.AI)){...}
        int idx = head;
        return new ScopeImpl(this, tag, idx, Time.realtimeSinceStartup);
    }

    public void NextFrame() {
        head = (head + 1) % FRAME_BUF; // 环形推进，旧数据自动覆盖
    }

    // 算某子系统的 P95（比平均值更能反映偶发卡顿）
    public float Percentile(Tag tag, float p) {
        var arr = new float[FRAME_BUF];
        for (int i = 0; i < FRAME_BUF; i++) arr[i] = frameCost[i, (int)tag];
        Array.Sort(arr);
        return arr[(int)(FRAME_BUF * p)]; // p=0.95 → P95
    }
}
// 使用：每帧用 using 块包裹子系统，自动计时入桶
```

**帧时间预算（Budget）模型：**

| 目标帧率 | 单帧预算 | 典型分配（参考） |
|----------|----------|------------------|
| 60fps | 16.6ms | 渲染 8ms / 逻辑 4ms / AI+物理 2ms / UI+IO 2ms |
| 30fps | 33.3ms | 渲染 16ms / 逻辑 8ms / 其他 9ms |

```
超预算处理流程：
  某子系统 P95 超预算 → 触发剖析 dump → 归因到具体函数
    → 是 GC？（降分配频率/对象池）
    → 是 DrawCall？（合批/LOD/剔除）
    → 是逻辑？（ECS 批处理/任务分帧/缓存）
```

**常用剖析工具对比：**

| 工具 | 定位 | 能力 | 适用阶段 |
|------|------|------|----------|
| Unity Profiler | CPU/GPU/内存 | 逐函数耗时、GC、内存对象 | 开发期 |
| RenderDoc | GPU 渲染 | 逐 DrawCall、Shader、纹理、帧捕获 | 开发期 |
| Xcode Instruments | iOS 系统级 | CPU 采样、内存、能耗、Metal | 开发+测试 |
| Android Studio Profiler | 安卓系统级 | CPU/内存/网络/能耗 | 开发+测试 |
| 自研/第三方 APM | 线上真实玩家 | 帧率分布、崩溃、卡顿热图 | 线上运营 |

### ⚡ 实战经验

- **盯 P95/P99，别盯平均值**：平均帧率 55fps 看着没问题，但 P95 掉到 20fps 意味着每秒都有一次明显卡顿，玩家体感极差。性能门禁要用分位数，平均值会骗人。
- **监控本身要有开关和限频**：上报 SDK 默认关闭或低频，按灰度比例开启，否则监控逻辑本身（序列化/网络上报）会拖慢低端机。崩溃时优先 dump 本地再异步上传，别在崩溃路径上做重活。
- **内存监控要区分"峰值"和"稳态"**：进战斗瞬间资源全加载是峰值（看会不会 OOM），长时间运行后的稳态水位看泄漏（看是否单调上涨）。GC 频率单独监控——频繁 GC 引起的卡顿比单次内存高更致命。
- **机型分桶是线上分析的核心**：旗舰机流畅 ≠ 全量流畅。按机型/芯片/内存档位分桶看帧率分布，优先修"覆盖玩家最多的中低端机型"的卡顿，而不是为 1% 的旗舰机优化到极致。
