## 参考答案

### ✅ 核心要点

1. **KCP 是纯算法层协议**：不负责底层收发，只在上层提供可靠传输语义，底层默认使用 UDP
2. **ARQ（自动重传请求）为核心**：通过序号 + 确认 + 超时重传实现可靠性，支持 Selective Repeat（选择性重传）
3. **快速重传不等超时**：收到 3 次重复 ACK 即触发重传，比 TCP 的阈值更激进
4. **可调节的可靠性—延迟权衡**：通过 `nodelay`、`interval`、`resend`、`nc` 四个参数灵活控制行为
5. **牺牲带宽换延迟**：KCP 的设计哲学是用 10%-20% 的额外带宽换取 30%-40% 的延迟降低

### 📖 深度展开

#### KCP 数据结构

```c
// KCP 核心数据段（简化）
struct IKCPSEG {
    uint32_t conv;       // 会话 ID
    uint16_t cmd;        // 命令类型: PUSH / ACK / WASK / WINS
    uint16_t frg;        // 分片序号（大消息拆分）
    uint32_t wnd;        // 可用窗口大小
    uint32_t ts;         // 时间戳
    uint32_t sn;         // 消息序号
    uint32_t una;        // 对方已收到的最小序号
    uint32_t len;        // 数据长度
    char data[];         // 负载数据
};
```

#### ARQ 重传机制对比

KCP 与 TCP 都使用 ARQ，但策略截然不同：

| 机制 | TCP | KCP（默认） | KCP（nodelay 模式） |
|------|-----|-------------|---------------------|
| 超时重传时间（RTO） | 1.5×SRTT | 1.5×SRTT | 1×SRTT |
| 超时退避 | 指数退避（×2） | 线性退避（×1.2） | 线性退避（×1.2） |
| 快速重传触发 | 3 次重复 ACK | 3 次重复 ACK | 可配置（默认 3） |
| 快速重传策略 | 降速 + 重传一个 | 不降速 + 重传多个 | 不降速 + 重传多个 |
| 选择性确认（SACK） | 可选支持 | 内置支持 | 内置支持 |

#### 快速重传的"重传多个"策略

TCP 的快速重传只重传最先丢失的那一个包。KCP 则不同——它会重传所有后续未被确认的包：

```
发送方发送: 1  2  3  4  5
接收方确认: 1  ✓  ✗  ✗  ✗  (2/3/4/5 丢失)

TCP 快速重传: 只重传 2
KCP 快速重传: 重传 2, 3, 4, 5（全部重传）
```

这看起来"浪费"带宽，但在游戏场景中，这些后续包大概率也已经丢了，与其等各自超时，不如一次全发。

#### 四个关键参数

```c
ikcp_nodelay(kcp, 1, 10, 2, 1);
// 参数说明：
// nodelay=1:  关闭常规延迟，RTO = SRTT（而非 1.5×SRTT）
// interval=10: 内部更新周期 10ms（越小越精确，CPU 越高）
// resend=2:   快速重传触发阈值 = 2 次重复 ACK（TCP 默认 3）
// nc=1:       关闭拥塞控制（不因丢包降速）
```

#### 拥塞控制取舍

KCP 默认关闭拥塞控制（`nc=1`），这是游戏场景的关键区别：

- **TCP 的拥塞控制**：丢包 → 降低发送速率 → 适合文件下载
- **KCP 无拥塞控制**：丢包 → 保持发送速率 → 适合实时游戏

在游戏场景中，"卡住不发包"比"丢几个包"对体验的伤害更大。玩家宁可看到一点卡顿（由应用层插值补偿），也不要完全无响应（TCP 队头阻塞）。

### ⚡ 实战经验

1. **conv ID 要匹配**：收发两端的 `conv` 必须一致，否则包会被直接丢弃。排查"数据发出去但收不到"的问题时，第一时间检查 conv
2. **大消息分片注意顺序**：KCP 的 `frg` 字段做消息拆分，接收端按逆序重组。如果你的游戏协议层也做了拆包，可能产生冲突，建议只在一层拆包
3. **interval 不要设太小**：在移动端，`interval=1ms` 会导致 CPU 占用飙升。建议手游项目用 `interval=10~20ms`
4. **监控 KCP 内部统计**：定期输出 `kcp->state` 中的等待确认段数、重传次数等指标，用于线上网络质量监控
