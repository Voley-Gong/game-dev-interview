## 参考答案

### ✅ 核心要点

| 维度 | 帧同步 | 状态同步 |
|------|--------|----------|
| **传输内容** | 只传操作指令 | 传完整游戏状态 |
| **带宽** | 低（每帧几个字节） | 高（状态数据量大） |
| **确定性要求** | 极高（浮点数必须一致） | 无要求 |
| **断线重连** | 需要重放所有帧 | 直接拉取当前状态 |
| **回放** | 天然支持 | 需额外实现 |
| **反作弊** | 弱（客户端算结果） | 强（服务端算结果） |
| **代表产品** | 王者荣耀、格斗游戏 | 原神、MMORPG |

### 📖 深度展开

**帧同步流程：**

```
客户端A          服务器          客户端B
  │               │               │
  ├─ input ──→    │               │
  │               ├─ lockstep ──→ │
  │    ←── broadcast ─┤          │
  │               │    ←── input ─┤
  │               │               │
  ▼               ▼               ▼
  [等待所有输入到达同一逻辑帧]
  │               │               │
  ▼               ▼               ▼
  [各客户端独立模拟，结果必须一致]
```

**状态同步流程：**

```
客户端A          服务器          客户端B
  │               │               │
  ├─ input ──→    │               │
  │               ├─ 计算 ──→     │
  │               ├─ state ──→    │
  │    ←── state ─┤    ←── state ─┤
  ▼               ▼               ▼
  [各客户端仅负责渲染和插值]
```

**帧同步的确定性挑战：**

```typescript
// ❌ 非确定性操作
Math.random();              // 随机数不一致
float a = 0.1 + 0.2;       // 浮点精度问题
list.sort();                // 排序不稳定
foreach (var item in dict)  // 遍历顺序不确定

// ✅ 解决方案
// 1. 使用确定性随机数（种子相同结果相同）
const rng = new SeededRandom(12345);
// 2. 使用定点数替代浮点数
const pos = new FixedPoint(100, 3); // 3位小数精度
// 3. 显式指定排序比较函数
list.sort((a, b) => a.id - b.id);
// 4. 使用有序数据结构
```

### ⚡ 实战经验

- **帧同步的坑**：确定性是最大的坑，不同设备/编译器的浮点行为可能不同
- **混合方案**：有些游戏用帧同步做战斗、状态同步做养成
- **网络抖动**：帧同步需要做延迟补偿和预测回滚（如 GGPO）
- **开发成本**：帧同步前期开发成本高，但带宽成本低、天然支持回放
- **选型建议**：竞技类（MOBA/格斗）选帧同步，RPG/MMO 选状态同步
