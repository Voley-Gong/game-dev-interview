## 参考答案

### ✅ 核心要点

| 协议 | 可靠性 | 延迟 | 有序性 | 适用场景 |
|------|--------|------|--------|----------|
| **TCP** | 可靠 | 较高（队头阻塞） | 有序 | 回合制、聊天、登录 |
| **UDP** | 不可靠 | 低 | 无序 | 实时位置、动作同步 |
| **KCP** | 可靠（ARQ） | 低 | 有序 | 实时对战（兼顾两者） |

### 📖 深度展开

**TCP 的问题：队头阻塞**

```
数据包发送序列：[1] [2] [3] [4] [5]
接收情况：      [1] ❌ [3] [4] [5]  (包2丢了)

TCP 行为：
  等待包2重传 → 包3/4/5在缓冲区等待 → 应用层收不到任何数据
  延迟 = 重传超时 + 处理延迟 ≈ 200ms-1s

游戏中这是致命的——玩家操作被阻塞。
```

**KCP 的解决方案：**

```
KCP = UDP + 可靠传输层（用户态实现）

核心机制：
1. 选择性重传（非 Go-Back-N）→ 只重传丢失的包
2. 快速重传（3个ACK触发）→ 不等超时就重传
3. 无 TCP 的拥塞控制 → 牺牲公平性换低延迟

// KCP 配置模式
// 普通模式：兼顾延迟和公平性
// 快速模式：极端低延迟，吞吐量换延迟
kcp.nodelay = true;       // 无延迟模式
kcp.interval = 10;        // 10ms 刷新间隔
kcp.resend = 2;           // 快速重传
kcp.noCongestion = true;  // 关闭拥塞控制
```

**实际游戏中的混合使用：**

```
┌──────────────────────────────────┐
│  网络层架构                       │
├──────────────────────────────────┤
│                                  │
│  ┌─ TCP ──→ 登录/充值/聊天      │
│  │                               │
│  ├─ KCP ──→ 战斗同步/操作指令    │
│  │                               │
│  └─ UDP ──→ 位置同步/语音        │
│                                  │
└──────────────────────────────────┘
```

### ⚡ 实战经验

- **KCP 在国内环境表现优秀**：弱网环境下比 TCP 延迟低 30-40%
- **UDP 被运营商 QoS 限速**：部分运营商会限速 UDP，需要 TCP fallback
- **WebSocket**：H5 游戏只能用 WebSocket（基于 TCP），如需 UDP 要用 WebRTC
- **帧同步推荐 KCP**：操作指令必须可靠送达，且延迟敏感
- **状态同步可混合**：位置用 UDP（丢一帧无所谓），关键操作用 KCP
