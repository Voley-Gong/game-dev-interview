## 参考答案

### ✅ 核心要点

1. **滑动窗口的本质是「维护一个在序列上移动的区间，用增量更新代替重复扫描」**：窗口右端扩展、左端收缩，每次只处理进出窗口的少量元素，从而把「对每个位置重新计算区间」的 O(n²) 压成 O(n)。关键是不重算整个区间，而是维护一个随窗口移动的不变量（和、计数、最值）。
2. **双指针是「两个下标单调前进」的通用技巧**：可以是同向（快慢指针、滑动窗口的左右边界）、也可以是相向（两端向中间）。游戏里最经典的同向双指针是**扫掠裁剪（Sweep & Prune）碰撞检测**：按 AABB 最小 x 排序后，用一个指针扫过，另一个标记可能重叠的范围，O(n + k) 找出所有潜在碰撞对。
3. **滚动均值/帧率平滑是滑动窗口的典型应用**：监控每帧耗时想求「最近 60 帧平均」，朴素做法每次对 60 帧求和是 O(60)；滑动窗口维护一个累加和，新帧加入减去滑出帧，O(1) 更新。配合环形缓冲还能零分配。
4. **滑动窗口 vs 前缀和 vs 暴力的取舍**：暴力每次重算 O(n²)；前缀和能 O(1) 答任意固定区间，适合「查询多、区间固定」；滑动窗口适合「区间本身要根据条件动态伸缩」。网络抖动缓冲（保留最近 N 个包做插值）就是窗口长度固定、内容滚动的场景，用环形缓冲实现。
5. **碰撞检测里双指针的精妙在于「利用排序的单调性剪枝」**：两个 AABB 在 x 轴不重叠就绝不可能碰撞。排序后，当 A 的右边界 < B 的左边界，A 之后的所有物体都更靠右，更不可能和 A 碰——直接 `break`。这让宽相位从 O(n²) 的两两检测降到接近 O(n + 实际重叠对数)。

### 📖 深度展开

#### 1. 扫掠裁剪碰撞检测（同向双指针）

```typescript
interface AABB { id: number; minX: number; maxX: number; }

// 单轴扫掠：找出 minX 上可能有重叠的 AABB 对，再做精确判定
function sweepAndPrune(boxes: AABB[]): [number, number][] {
  boxes.sort((a, b) => a.minX - b.minX);          // 按 minX 排序（O(n log n)）
  const pairs: [number, number][] = [];
  for (let i = 0; i < boxes.length; i++) {
    for (let j = i + 1; j < boxes.length; j++) {
      // ★ 单调性剪枝：j 的 minX >= i 的 minX，若 i 的 maxX < j 的 minX
      //   则 j 及之后都不可能与 i 重叠 → break
      if (boxes[i].maxX < boxes[j].minX) break;
      pairs.push([boxes[i].id, boxes[j].id]);     // 潜在碰撞对，交窄相位精确判定
    }
  }
  return pairs;
}
// 物体稀疏时内层循环几乎立即 break → 接近 O(n)；密集时退化为 O(n²)
// 增量优化：帧间物体移动小，维护「已排序+活动区间」可摊到 O(n + 变化量)
```

```
扫掠裁剪示意（按 minX 排序后单轴扫掠）：

  AABB:   [A]    [B]  [C]     [D]
  minX:    1      5    8       20
  maxX:    4      9    12      24
           │             │      │
   i=A:  A.maxX(4) < B.minX(5)? 否 → 配对(A,B)；< C.minX(8)? 否 → (A,C)；
         A.maxX(4) < D.minX(20)? 是 → break（D 及之后都不碰 A）
   i=B:  B(9) 与 C(8..12) 重叠 → (B,C)；< D(20)? 是 → break
   结果：候选对 {(A,B),(A,C),(B,C)}，D 独处无碰撞 —— 远少于 n² 全配对
```

#### 2. 滚动均值的滑动窗口（环形缓冲零分配）

```typescript
// 滑动窗口维护最近 N 帧的累加和，O(1) 更新均值
class RollingAverage {
  private buf: number[];          // 环形缓冲
  private head = 0;               // 下一个写入位置
  private sum = 0;                // ★ 窗口不变量：当前窗口内元素之和
  private filled = 0;             // 已填入数量（启动期 < N）

  constructor(private n: number) { this.buf = new Array(n).fill(0); }

  push(v: number): void {
    this.sum -= this.buf[this.head];   // 减去即将被覆盖的旧值（滑出窗口）
    this.buf[this.head] = v;           // 写入新值
    this.sum += v;                     // 加入新值
    this.head = (this.head + 1) % this.n;  // 环形前进，零分配
    if (this.filled < this.n) this.filled++;
  }

  average(): number {
    return this.filled === 0 ? 0 : this.sum / this.filled;
  }
}
// 用途：监控帧耗时、网络 RTT 平滑、DPS 窗口统计、音频电平表
const fps = new RollingAverage(60);
eachFrame(dt => fps.push(dt));    // O(1) 更新，60 帧均值实时可得
```

#### 3. 网络抖动缓冲：固定窗口 + 内容滚动

```
客户端收到服务器位置包，按到达顺序放入抖动缓冲（滑动窗口），插值渲染：

时间 →  t0   t1   t2   t3   t4   t5   t6
包:    P0   P1   P2   P3   P4   P5   P6
                              ↑ 渲染读取点（落后窗口 100ms 抹平抖动）
        └────── 滑动窗口（保留最近 N 个包做插值）──────┘

  新包到达 → 入队尾部；渲染后 → 弹出头部
  窗口长度固定（如 100ms），内容随时间滚动
  乱序/丢包：窗口内插值/外推平滑，玩家感知不到网络抖动
```

| 场景 | 窗口类型 | 维护的不变量 | 复杂度 | 关键技巧 |
|------|---------|-------------|--------|---------|
| 扫掠裁剪碰撞 | 双指针区间 | x 轴重叠关系 | O(n+k) | 排序 + break 剪枝 |
| 滚动帧率均值 | 定长滑动 | 区间元素和 | O(1) 更新 | 环形缓冲 |
| 网络抖动缓冲 | 定长滚动 | 时序包队列 | O(1) 入出 | 插值/外推 |
| 战斗 DPS 统计 | 时间窗 | 窗口内伤害和 | O(事件数) | 时间戳滑出剔除 |
| 范围技能选取 | 双指针/树 | 区间内目标集 | O(n) / O(log n) | 与空间索引配合 |

### ⚡ 实战经验

- **扫掠裁剪排序每帧重排拖垮性能**：200 个敌人每帧 `sort` 是 O(n log n) 还带分配，低端机明显掉帧。利用「帧间物体移动很小」，维护上一帧的有序数组，每帧用插入排序微调（近似 O(n)），排序开销降 80%。物体瞬移（传送）时要降级重排。
- **滚动均值启动期没处理「未填满」导致显示异常**：`RollingAverage` 初始 buffer 全 0，前 60 帧均值被一堆 0 拉低，FPS 显示从 0 缓慢爬升。加 `filled` 计数，启动期只除以已填数量后立刻准确。
- **网络窗口太小卡顿、太大延迟**：抖动缓冲窗口设 50ms 时网络稍抖就丢包卡顿，设 300ms 又手感发黏。实测在目标 RTT 的 1.5~2 倍（如 80ms 抖动用 150ms 窗口）较平衡，且要做自适应：抖动稳定时收窄窗口降延迟。
- **DPS 窗口用数组每次 filter 每帧 O(n) 爆炸**：统计「过去 5 秒伤害」最初每帧 `damageLog.filter(t => t > now-5000)`，战斗日志几千条时卡顿。改用双指针/链表头尾（头部压入新伤害、弹出过期头部），每帧只处理边界，开销恒定。
